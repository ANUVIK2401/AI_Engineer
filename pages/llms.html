What Are LLMs — Scale, Parameters & Emergent Behaviour

A Large Language Model is a Transformer trained to predict the next token on internet-scale text. "Large" refers to parameter count — measured in billions — but the more surprising finding is that scale doesn't just make models better at known tasks, it unlocks entirely new capabilities that smaller models cannot do at all.

Scale of modern LLMs — parameters vs capability
Emergent capabilities — appear suddenly above scale thresholds
Chain-of-thought reasoning
In-context few-shot learning
Instruction following
Multi-step arithmetic
Code generation
Language translation
Theory of mind
Calibrated uncertainty

Self-supervised pretraining

LLMs are trained on next-token prediction across trillions of tokens. No labels — the text itself provides the supervision signal. Scale + diversity = generalisation.

Parameters as memory

Each parameter is a weight storing a learned pattern. LLaMA-3-70B has 70 billion — more parameters than synapses in 7 human brains combined.

Chinchilla scaling laws

Compute should be split equally between model size and token count. GPT-3 was undertrained — LLaMA-3 trained on 15T tokens to fix the imbalance.

💡
Emergence is not fully understood Emergent capabilities don't appear gradually — they phase-transition. A model at 50B parameters may score 0% on a benchmark; a model at 70B scores 85%. Whether this reflects discontinuous generalisation or a measurement artefact is an active research debate.
🔧
Why an AI Engineer needs this Model selection comes down to scale vs cost. A 7B model serves 50× more requests per dollar than a 70B model. If your task is in the 7B capability range, don't use 70B. If it requires emergent reasoning — multi-step planning, code debugging, complex analysis — you need a frontier model. Benchmark on your specific task before choosing.

Tokenisation — Interactive Demo

LLMs don't read characters or words — they read tokens. A tokeniser splits text into subword units using Byte-Pair Encoding (BPE). Common words become single tokens; rare or technical words split into several. This has direct cost and context window implications.

How BPE works

Start with individual bytes. Repeatedly merge the most frequent adjacent pair into a new token. After 50k–100k merges you have a vocabulary. Common sequences ("the", "ing", "tion") become single tokens; rare ones split.

Tokenisation quirks

Whitespace is included in tokens. "dog" ≠ " dog". Numbers often split digit-by-digit. Non-Latin scripts use more tokens per character — Arabic and Chinese cost 3–5× more than English per semantic unit.

Why an AI Engineer needs this API costs are billed per token, not per character. A verbose system prompt uses 500 tokens before the user speaks. Knowing tokenisation lets you estimate costs, optimise system prompts, and understand why multi-digit arithmetic, URL parsing, and code completion are harder than they look — token boundaries misalign with semantic boundaries.

Prompt Engineering — Zero-shot, Few-shot & Chain-of-Thought

How you structure a prompt determines output quality more than which model you use. Select a technique below to see a before/after example rendered live — bad prompt on the left, improved prompt on the right.

Select a technique to see it in action
Without technique
Classify the sentiment. Text: "The hotel room was small but the location was perfect and the staff were incredibly friendly."
Positive
Oversimplified. The review is mixed — the model collapsed nuance into a single label without being asked for detail.
With zero-shot instruction
You are a sentiment analysis expert. Classify as Positive, Negative, or Mixed and explain which aspects drove each label. Text: "The hotel room was small but the location was perfect and the staff were incredibly friendly."
Sentiment: Mixed • Negative: "room was small" • Positive: "location was perfect", "staff incredibly friendly" Overall lean: Positive (2 strong positives vs 1 minor negative)
Role + explicit format + reasoning request → structured, nuanced output with no examples needed.
💡
Zero-shot principle No examples — just a clear instruction. Specify: (1) the model's role, (2) exact output format, (3) any constraints. Works well for common tasks; fails on novel schemas or domain-specific output formats.
Zero-shot for novel format
Extract product info from reviews. Review: "Bought the Sony WH-1000XM5 headphones. Paid $350. Noise cancellation is incredible but ear cups get warm after 2h."
The product is Sony WH-1000XM5 headphones. Price was $350. Reviewer liked noise cancellation but found ear cups warm.
Free-form prose. No consistent schema — not parseable programmatically.
With few-shot examples
Extract product info as JSON. Review: "Tried Bose QC45 for $279. Great comfort, mediocre battery." Output: {"product":"Bose QC45","price":279,"pros":["comfort"],"cons":["battery"]} Review: "Samsung 4K TV, $799. Stunning picture but cheap remote." Output: {"product":"Samsung 4K TV","price":799,"pros":["picture"],"cons":["remote"]} Review: "Bought Sony WH-1000XM5 headphones. Paid $350. Noise cancellation is incredible but ear cups get warm after 2h." Output:
{"product":"Sony WH-1000XM5","price":350,"pros":["noise cancellation"],"cons":["ear cups warm after 2h"]}
Two examples taught the exact schema. Output is valid JSON, parseable without post-processing. Format shown > format described.
💡
Few-shot principle 2–8 input/output examples demonstrate the exact format and style you want. Critical for: custom JSON schemas, specific writing styles, classification with unusual label sets. Keep examples diverse — don't repeat the same scenario.
Direct answer prompt
A factory produces 240 widgets/hour, runs 16h/day, 5 days/week. Widgets sell for $4.50, cost $0.80 each, plus $8,000 fixed daily costs. What is the weekly profit?
$14,400
Wrong. The model skipped intermediate steps and made arithmetic errors. Direct-answer prompts on multi-step math are unreliable.
Chain-of-thought prompt
Solve step by step, showing each calculation. A factory produces 240 widgets/hour, runs 16h/day, 5 days/week. Widgets sell for $4.50, cost $0.80 each, plus $8,000 fixed daily costs. What is the weekly profit?
Step 1: Weekly widgets 240 × 16 × 5 = 19,200 Step 2: Revenue 19,200 × $4.50 = $86,400 Step 3: Variable costs 19,200 × $0.80 = $15,360 Step 4: Fixed costs $8,000 × 5 = $40,000 Step 5: Profit $86,400 − $15,360 − $40,000 = $31,040
Serialised reasoning forces the model to use each step as context for the next. Accurate, auditable, and correct.
💡
Chain-of-thought principle "Think step by step" dramatically improves multi-step reasoning and math. Generated intermediate tokens act as a scratchpad — each step becomes context for the next. This is the mechanism behind extended thinking and o1-style reasoning models.
🔧
Why an AI Engineer needs this Prompt engineering is the highest-leverage, lowest-cost lever you have. Before spending on fine-tuning or RAG infrastructure, exhaust: (1) system prompt role + constraints, (2) output format specification, (3) few-shot demonstrations, (4) chain-of-thought for reasoning. A well-crafted prompt on a 7B model often beats a naive prompt on 70B — and costs 10× less.

Context Window — How Tokens Fill Memory

The context window is the total number of tokens a model can "see" at once — system prompt, conversation history, retrieved documents, and the current user message all share this fixed budget. Running out means the model silently truncates the oldest content.

Typical context composition — 128k window, RAG chatbot

When the window fills, older messages are truncated — the model forgets them silently.

💡
Lost-in-the-middle problem Studies show LLMs recall information at the beginning and end of context reliably, but memory degrades for content buried in the middle. Put critical information last, closest to the question.
🔧
Why an AI Engineer needs this Latency and cost scale with context length. Design your context budget explicitly: cap system prompts, summarise old history instead of appending, truncate RAG chunks to the minimum needed. Large context windows are a last resort, not a default.

Temperature & Sampling — Controlling Randomness

LLMs don't pick the single best token — they sample from a probability distribution. Temperature controls how "peaked" or "flat" that distribution is. Top-p and top-k limit which tokens are eligible. Together they determine whether your model gives reliable facts or creative surprises.

Interactive — drag temperature to see how the distribution changes
Prompt: "The capital of France is" — next token probabilities:
T = 0.70
Softmax with temperature T
$$P(x_i) = \frac{e^{z_i / T}}{\sum_j e^{z_j / T}}$$

T→0: deterministic (argmax). T=1: raw model probabilities. T>1: flattened → high randomness.

ParameterWhat it doesTypical rangeBest for
TemperatureScales all logits before softmax0–1 prod; up to 2 creativeGlobal creativity dial
Top-p (nucleus)Sample only from tokens summing to p probability mass0.85–0.95Prevents incoherent long-tail tokens
Top-kConsider only k highest-probability tokens40–100Hard cap on vocabulary size
Repetition penaltyReduces logits of recently seen tokens1.0–1.3Prevents loops in open-ended generation
🔧
Production defaults by use case Factual Q&A / code: T=0–0.2, top-p=0.9. Chatbots / assistants: T=0.5–0.7. Creative writing / brainstorming: T=0.9–1.2, top-p=0.95. Never use T=0 for chat — identical responses feel robotic.

RLHF — Reinforcement Learning from Human Feedback

Base LLMs predict the next token — they don't naturally follow instructions or avoid harmful outputs. RLHF is the three-stage pipeline that turned raw GPT-3 into ChatGPT, using human preferences to align model behaviour with human intent.

Three-stage RLHF pipeline
1
Supervised Fine-Tuning (SFT)
Human labellers write ideal responses to thousands of prompts. The base model is fine-tuned on these (prompt, ideal-response) pairs using standard cross-entropy loss. This teaches instruction-following format and safe response patterns.
↓
2
Reward Model (RM) Training
For each prompt the SFT model generates multiple responses. Humans rank them pairwise. A separate Reward Model — another LLM with a scalar output head — is trained to predict human preference scores from these comparisons, distilling human judgement into a differentiable function.
↓
3
PPO Optimisation (RL phase)
The SFT model is the RL agent. It generates responses, the Reward Model scores them, and PPO updates the model to maximise expected reward. A KL-divergence penalty prevents the model from drifting too far from the SFT base — without it, the model would exploit the reward model with high-scoring gibberish.

DPO — the modern alternative

Direct Preference Optimisation (2023) achieves RLHF-quality alignment without a separate reward model or RL loop. It directly optimises the policy on preference pairs. Much simpler to implement, more stable. Used in LLaMA-3, Mistral Instruct, and most modern open models.

Constitutional AI (Claude)

Instead of human preference labels, the model is given a "constitution" of principles and trained to critique and revise its own outputs. Reduces dependence on labellers at scale. Combines SFT + RLAIF (AI feedback) rather than pure human feedback.

💡
Reward hacking — Goodhart's Law If the reward model is imperfect (it always is), PPO can exploit its blind spots — generating responses that score high on the proxy reward but aren't actually helpful. The KL penalty and careful reward model design are the main mitigations.
🔧
Why an AI Engineer needs this When your fine-tuned model gives technically correct but weirdly formatted or off-tone responses, that's a symptom of SFT-only training without preference data. Adding DPO with even 500 preference pairs can dramatically improve output quality. Understanding RLHF also explains why base models (LLaMA-3-base) are hard to use directly — they predict tokens, not answers.

Embeddings — Vector Space & Word Relationships

Embeddings are dense vector representations of text where semantic similarity corresponds to geometric proximity. Words, sentences, or documents map to points in a high-dimensional space (1536D for OpenAI's text-embedding-3-small). The 2D projection below shows how semantic clusters emerge naturally from meaning.

Word analogy — vector arithmetic in embedding space
$$\text{king} - \text{man} + \text{woman} \approx \text{queen}$$

Relationships encode as vector offsets: gender, royalty, location, and sentiment are directions in embedding space.

Cosine similarity

Similarity = cosine of the angle between two vectors. Range: −1 (opposite) to +1 (identical). Used by every vector database for nearest-neighbour search. Dot product is faster when vectors are normalised (L2 norm = 1).

Embedding models

text-embedding-3-small (OpenAI): 1536D, fast, cheap. text-embedding-3-large: 3072D, better recall. BGE-M3 (open): multilingual, 8k context. sentence-transformers: local, flexible. Choose based on latency vs quality vs cost.

python 
from openai import OpenAI
import numpy as np

client = OpenAI()

def embed(text: str) -> np.ndarray:
    resp = client.embeddings.create(
        model="text-embedding-3-small",
        input=text
    )
    return np.array(resp.data[0].embedding)

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))
