01Beginner

The ML Lifecycle — Data → Train → Evaluate → Deploy → Monitor

MLOps is what happens between a Jupyter notebook and a model that's reliably serving users six months later. The ML lifecycle is a loop, not a line: every production deployment generates new data and feedback that drives the next training iteration.

ML Lifecycle — continuous loop diagram

Why it's a loop

Production data reveals distribution shifts, edge cases, and label drift that notebooks never see. Every deployment is a data collection experiment that improves the next model version.

Model registry

A versioned catalogue of trained model artifacts with stage labels (Staging → Production → Archived). MLflow Registry and W&B Artifacts are common implementations. Enables rollback in under 60 seconds.

Canary deployment

Route 5–10% of traffic to the new model version, compare metrics against the old version in production, then gradually increase if metrics hold. Cuts rollout risk dramatically.

💡
The notebook-to-production gap A model that achieves 94% accuracy in a notebook can behave catastrophically in production. The notebook uses a fixed, clean dataset; production sees real-time, messy, drifting data. MLOps exists to close this gap systematically.

02Intermediate

Model Serving — REST, Batch, Streaming & the Latency vs Throughput Tradeoff

How you serve a model is as important as the model itself. A fraud detector needs a response in under 50ms during a payment; a churn prediction model can score overnight. Matching serving mode to use case determines infrastructure cost, latency, and complexity.

Online / REST

Synchronous API

Client sends request, waits for prediction. FastAPI + uvicorn is the standard Python stack. Best when users or systems need the result immediately.

latency goal <100ms fraud · rec · search
Batch

Offline Scoring

Score millions of records in bulk during off-peak hours. Results written to database, then read by downstream systems. No latency requirement — throughput is everything.

throughput-first churn · risk scores
Streaming

Event-Driven

Model consumes a Kafka or Kinesis stream, scores each event, and publishes predictions to another topic. Continuous low-latency scoring without per-request overhead.

continuous ad-bidding · IoT

Dynamic batching: accumulate requests for a short window (5–10ms) then process as a batch. Most serving frameworks (Triton, TorchServe, vLLM) do this automatically.

Instrument p50/p95/p99 from day one; averages hide the tail.

Containerization — Docker for ML Models

Docker solves the "works on my machine" problem that plagues ML deployments. A container bundles your Python environment, CUDA version, model weights, and serving code into a single portable image that runs identically from a laptop to a Kubernetes cluster.

What's inside a Docker image for an ML model
Dockerfile # ── Stage 1: install heavy deps ───────────────────── FROM python:3.11-slim AS builder WORKDIR /app COPY requirements.txt . # Wheel-only installs avoid build deps in the final image RUN pip install --no-cache-dir --user -r requirements.txt # ── Stage 2: slim runtime image ───────────────────── FROM python:3.11-slim WORKDIR /app # Copy only installed packages (not build tools) COPY --from=builder /root/.local /root/.local # Copy model weights from a local registry mount or ARG COPY model.onnx . COPY serve.py . ENV PATH=/root/.local/bin:$PATH EXPOSE 8080 HEALTHCHECK --interval=30s CMD curl -f http://localhost:8080/health || exit 1 CMD ["uvicorn", "serve:app", "--host", "0.0.0.0", "--port", "8080"]

Layer caching strategy

Order layers by change frequency: base image (never) → deps (weekly) → model weights (per release) → app code (daily). A changed layer invalidates all layers above it.

Multi-stage builds

Compilers and build tools inflate image size. Build in a fat stage, then COPY only the artifacts into a slim runtime. Final image can be 5× smaller with the same functionality.

GPU containers

Use nvidia/cuda:12.x-cudnn-runtime as base. Add --gpus all or the NVIDIA device plugin in Kubernetes. Runtime images are much smaller than devel images.

💡
Model weights don't belong in the image Baking large weights (1–70GB) directly into the image makes builds slow and pushes to a registry painful. Instead, pull weights from S3/GCS at container startup, or mount them as a volume. The image should be <2GB so it builds and deploys in under 3 minutes.

CI/CD for ML — How ML Pipelines Differ from Software CI/CD

Traditional CI/CD checks that code compiles and tests pass. ML CI/CD must also validate that the trained model meets quality thresholds — a pipeline that trains a worse model than what's in production should never deploy, even if the code is syntactically correct.

The Model Gate (ML-specific)

After training, compare the new model against the current production model on a held-out dataset. If the challenger underperforms on key metrics (AUC, F1, NDCG), the pipeline fails and deployment is blocked — just like a failing test in software CI.

CML — CI for ML

CML (Continuous Machine Learning) by Iterative.ai adds ML-aware steps to GitHub Actions: post model metrics as PR comments, generate performance plots, and compare experiments. Makes model quality visible in code review.

YAML # .github/workflows/ml-pipeline.yml name: ML Train-and-Deploy on: push: paths: ["src/**", "data/schema/**"] jobs: train: runs-on: ubuntu-latest # use self-hosted GPU runner for real training steps: - uses: actions/checkout@v4 - name: Train model run: python src/train.py --config configs/prod.yaml env: {MLFLOW_TRACKING_URI: ${{ secrets.MLFLOW_URI }}} - name: 🚦 Model gate — fail if AUC drops run: | python src/evaluate.py \ --challenger runs/latest/model.pkl \ --champion models/production/model.pkl \ --min-auc 0.85 \ --min-improvement 0.005 # must beat champion by 0.5% - name: Build & push Docker image if: success() run: | docker build -t $ECR_REPO:${{ github.sha }} . docker push $ECR_REPO:${{ github.sha }} - name: Deploy to staging (canary 5%) if: success() run: ./scripts/deploy.sh --env staging --weight 5

05Intermediate

Model Monitoring — Data Drift, Concept Drift & Performance Degradation

A model that was 94% accurate at launch can silently degrade to 71% accuracy six months later without a single line of code changing. Production data drifts — user behaviour evolves, data pipelines change schema, seasonal patterns shift. Monitoring catches this before users notice.

Data Drift — P(X) shifts
Input feature distribution changes
New user segments, seasonal behaviour
Upstream schema change in ETL
Detection: PSI, KS test, chi-squared
Can detect without labels
Concept Drift — P(Y|X) shifts
Relationship between inputs and outputs changes
Fraud patterns evolve, preferences change
World events alter true labels (COVID)
Detection: requires ground-truth labels
Harder — often has days/weeks of delay
PSI — Population Stability Index (the standard data drift score)
● PSI < 0.1: stable, no action ● PSI 0.1–0.2: monitor closely ● PSI > 0.2: investigate + consider retraining

What to monitor

Input feature distributions (PSI per feature), prediction score distribution, model performance (if labels available), output confidence, and request latency.

Shadow mode

Run a challenger model in parallel without serving its predictions to users. Compare distributions against the champion. Safe way to validate a new model on live data before cutover.

Alert thresholds

Page on: PSI > 0.2, accuracy drops >2% week-over-week, p99 latency >200ms, prediction distribution skews 20%+. Set per-feature thresholds, not just global ones.

06Intermediate

Observability — What to Log, Prometheus & Grafana

You cannot monitor what you don't measure. Observability for ML systems goes beyond infrastructure health (CPU/memory) to include prediction-level logging, latency histograms, and model-specific signals. Without it, degradation is invisible until users complain.

Oracle context: Our fraud detection stack used Prometheus + Grafana dashboards displaying p50/p95/p99 prediction latency, prediction score distribution by hour, and daily PSI per top-10 feature. One dashboard saved us from a silent model failure after an upstream ETL pipeline changed a categorical encoding.

The Four Golden Signals (ML edition)

Latency — p50/p95/p99 per endpoint. Don't track averages alone.
Traffic — requests/sec, batch throughput per model version.
Errors — HTTP 5xx, timeouts, malformed prediction outputs.
Saturation — GPU utilisation, memory pressure, queue depth.

ML-specific signals

Prediction score distribution — histogram of model outputs. A shift here often precedes accuracy degradation by days.
Feature value distributions — PSI per feature, alerting on drift.
Confidence entropy — model becoming more uncertain signals concept drift.

Python from prometheus_client import Counter, Histogram, Gauge, generate_latest import time, numpy as np # ── Metric definitions ────────────────────────────── PREDICTION_COUNT = Counter("model_predictions_total", "Total predictions", ["model_version", "result_bucket"]) PREDICTION_LATENCY= Histogram("model_prediction_latency_seconds", "Prediction latency", buckets=[.005, .01, .025, .05, .1, .25, .5, 1]) SCORE_HISTOGRAM = Histogram("model_score_distribution", "Prediction score distribution", buckets=[i/10 for i in range(11)]) GPU_UTILISATION = Gauge("gpu_utilisation_pct", "GPU utilisation %") # ── Instrumented predict endpoint ─────────────────── @app.post("/predict") async def predict(req: PredictRequest): start = time.perf_counter() score = float(model.predict(req.features)) latency = time.perf_counter() - start PREDICTION_LATENCY.observe(latency) SCORE_HISTOGRAM.observe(score) bucket = "high" if score > .7 else "medium" if score > .3 else "low" PREDICTION_COUNT.labels(model_version="v2.1", result_bucket=bucket).inc() # ── Structured prediction log (JSON to stdout → log aggregator) import json, logging logging.info(json.dumps({ "event": "prediction", "model_version": "v2.1", "score": round(score, 4), "latency_ms": round(latency*1000, 2), "feature_hash": hash(str(req.features)) % 10**8 # privacy-safe })) return {"score": score} @app.get("/metrics") # Prometheus scrapes this endpoint every 15s async def metrics(): return generate_latest()

Prometheus scraping

Prometheus polls /metrics every 15–60 seconds. Each service exposes counters, gauges, and histograms in the text format. Alertmanager fires on Prometheus rules.

Grafana dashboards

Query Prometheus with PromQL. Build panels for: latency percentiles, request rate, score distribution over time, drift alerts, cost (tokens × price for LLMs).

What to log per prediction

Model version, timestamp, latency, input feature hash (not raw features for privacy), output score/decision, and a request ID. Store logs in a data warehouse for retrospective analysis.

Quantization & Inference Optimization — INT8, FP16, GGUF, vLLM, Speculative Decoding

A 70B parameter model in FP32 needs 280GB of GPU memory. The same model in INT4 fits on two consumer GPUs. Quantization and inference optimizations are the bridge between "this model is impressive" and "this model is deployable."

GGUF — CPU Inference

GGUF (GPT-Generated Unified Format) is used by llama.cpp for 4-bit quantized inference on CPUs and Apple Silicon. Enables running LLaMA-3-70B on a MacBook Pro M3 Max (~36GB unified memory) without any GPU. The go-to format for local development and edge deployments.

vLLM — PagedAttention

vLLM's PagedAttention manages KV-cache memory like an OS manages virtual memory — no wasted pre-allocated blocks. Result: 24× higher throughput than naive HuggingFace serving with the same hardware. Use vLLM for any production LLM endpoint handling more than a few concurrent users.

Speculative decoding — how it achieves 2–3× speedup

Flash Attention

Rewrites attention computation to avoid materialising the full N×N attention matrix in HBM memory. 2–4× faster attention, 10–20× less memory for long sequences. Now standard in all serious LLM serving.

ONNX & TensorRT

ONNX provides a universal IR for exporting models from any framework. TensorRT (NVIDIA) JIT-compiles ONNX graphs into GPU-optimised kernels — typical 2–4× speedup for CV and NLP models on NVIDIA hardware.

When to quantize

INT8 for classic ML (negligible accuracy loss). BF16 for LLMs in production. INT4/GGUF for local/edge deployments. Never quantize the final classification head — it needs full precision. Benchmark on your eval set before and after.

Cloud AI Services — AWS SageMaker, GCP Vertex AI, Azure ML

Cloud ML platforms abstract away the infrastructure layer so teams focus on models rather than Kubernetes manifests. The tradeoff: managed convenience vs flexibility and vendor lock-in. Understanding what each platform provides — and where they differ — helps you choose the right tool and avoid being surprised by the bill.

When to use managed

You want to move fast and don't have a dedicated MLOps team. Managed services handle autoscaling, load balancing, health checks, and container orchestration. Great for startups and small teams.

When to go custom

You need specific hardware (H100 clusters), have bespoke serving requirements, or need to avoid vendor lock-in. Self-managed Kubernetes with Kubeflow gives maximum flexibility at the cost of operational complexity.

Cost reality

Cloud GPU instances are expensive. ml.p3.2xlarge (V100) on SageMaker: ~$3.83/hr. Use spot/preemptible for training (fault-tolerant jobs only). Use serverless endpoints for low-traffic models; they scale to zero when idle.
