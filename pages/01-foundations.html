AI vs ML vs DL vs GenAI

[FREQUENTLY ASKED]
  • AI is the broad goal of building systems that perform tasks requiring human-like intelligence; ML is a subset that learns patterns from data instead of following hardcoded rules; DL is a subset of ML using layered neural networks; GenAI is a subset of DL focused on generating new content rather than just predicting labels.
  • This hierarchy exists because each layer solves a limitation of the one above it — ML replaced brittle rule engines, DL replaced manual feature engineering, GenAI replaced discriminative-only outputs with generative ones.
  • The insight: every GenAI system is still just ML underneath — the "magic" is scale and objective (next-token prediction), not a new paradigm. Someone who "gets it" can explain an LLM in terms of loss functions and gradients, not buzzwords.
▸ How this shows up in interviews
"Walk me through the difference between AI, ML, and deep learning."
"AI is robots, ML is when computers learn, deep learning is neural networks."
Each is a strict subset of the last, defined by how the mapping from input to output is obtained. AI includes hand-coded rule systems that never learn anything. ML restricts that to systems that fit parameters from data — but classic ML still relies on hand-engineered features. Deep learning removes that requirement by learning hierarchical representations directly, which is why it scales so well with data and compute. Generative AI is deep learning applied to modeling a full data distribution (e.g., next-token or next-pixel) so it can sample new outputs, not just classify.
Follow-up: "Give an example task where classic ML would beat deep learning."

Vectors and Embeddings

[CRITICAL]
  • A vector is an ordered list of numbers representing a point in space; an embedding is a learned vector where geometric distance encodes semantic similarity.
  • Embeddings exist because raw data (words, images, users) isn't naturally numeric or comparable — you need a representation where "similar things are close together" so math (dot products, distances) becomes meaningful.
  • The insight: embedding spaces are learned, not designed — the geometry (what "close" means) emerges entirely from the training objective, so two embedding models trained on different tasks are not interchangeable or comparable, even at the same dimensionality.
▸ How this shows up in interviews
"What makes an embedding 'good'?"
"It has a high dimension so it captures more information."
A good embedding is one where distance in the vector space correlates with task-relevant similarity — not dimensionality for its own sake. Higher dimensions can capture finer distinctions but also cost more memory and search latency, and past a point add noise rather than signal (curse of dimensionality). Quality is measured empirically against a downstream task or benchmark (like MTEB for text), because embeddings are only "good" relative to what you use the similarity for.
Follow-up: "Why can't you mix embeddings from two different models in one vector index?"

Matrix Multiplication & Dot Products

[CRITICAL]
  • The dot product of two vectors measures how much they point in the same direction — high when aligned, near zero when orthogonal, negative when opposed.
  • It matters because nearly every neural network operation (linear layers, attention scores, similarity search) is fundamentally "how much does this input align with this learned direction."
  • The insight: attention is just dot products between queries and keys, scaled and softmaxed — if you understand dot product as "similarity score," multi-head attention stops looking like magic and becomes "weighted average based on relevance."
$$ \mathbf{a} \cdot \mathbf{b} = \sum_{i=1}^n a_i b_i = |\mathbf{a}||\mathbf{b}|\cos\theta $$
▸ How this shows up in interviews
"Why does attention use a dot product between Q and K instead of, say, Euclidean distance?"
"Because that's how the paper defined it."
Dot product is cheap — a single matrix multiplication computes all pairwise scores at once, which is critical given GPUs are optimized for matmul throughput. It also naturally captures directional alignment, which is what "relevance" means in a learned embedding space, whereas distance metrics conflate magnitude and direction in ways that complicate learning. In practice, some variants (additive attention) do use other scoring functions, but scaled dot-product wins because it's both expressive and hardware-friendly.
Follow-up: "What's the computational complexity of computing all pairwise attention scores for a sequence of length n?"

Probability Essentials

[FREQUENTLY ASKED]
  • Conditional probability P(A|B) is the likelihood of A given that B already happened; Bayes' theorem lets you flip that around to infer P(B|A) from known priors and likelihoods.
  • LLMs exist because language modeling is defined as learning P(next token | previous tokens) — the entire training objective is a probability distribution over vocabulary, nothing more.
  • The insight: "hallucination" isn't a bug bolted onto a fact-retrieval system — it's the expected behavior of a model whose only job is to sample from a learned probability distribution. There is no separate "truth check" layer unless you build one.
$$ P(A \mid B) = \frac{P(B \mid A)\,P(A)}{P(B)} $$
▸ How this shows up in interviews
"In what sense is an LLM 'just' a probability distribution?"
"It predicts the next word based on patterns it saw in training."
Formally, the model parameterizes P(token_t | token_1...token_{t-1}) over the entire vocabulary at every position, trained via maximum likelihood (cross-entropy) on next-token prediction. Generation is sampling from that conditional distribution, repeatedly, autoregressively — which is exactly why decoding strategy (temperature, top-p) meaningfully changes output even with identical model weights. This framing also explains why RLHF works: it reshapes the distribution's shape (via a reward signal) rather than adding any new "reasoning module."
Follow-up: "If it's just a distribution, why does chain-of-thought prompting improve accuracy on reasoning tasks?"

Gradient Descent

[CRITICAL]
  • Gradient descent iteratively updates parameters in the direction that most reduces the loss, using the gradient (slope) of the loss function with respect to each parameter.
  • It exists because most models have no closed-form solution for their optimal parameters — the loss landscape is too complex, so we search it numerically instead.
  • The insight: the learning rate isn't just a speed knob — it determines whether you converge at all. Too high and updates overshoot and diverge; too low and you get stuck in flat regions or take impractically long; this is why LR schedules and warmup exist.
$$ \theta_{t+1} = \theta_t - \eta \nabla_\theta \mathcal{L}(\theta_t) $$
Situation
What breaks or costs
Learning rate too high
Loss oscillates or diverges instead of converging
Deep, narrow local minima / saddle points
Training stalls; gradients near zero make progress slow
Very large batch size
Smoother gradients but worse generalization and higher memory cost
▸ How this shows up in interviews
"Why do we use stochastic/mini-batch gradient descent instead of full-batch?"
"It's faster because you use less data per step."
Mini-batches make each update cheap enough to fit in memory and give noisy gradient estimates that actually help escape sharp local minima and saddle points, which full-batch gradient descent gets stuck in more easily. There's a real tradeoff though: too small a batch adds so much noise that convergence becomes unstable, while too large a batch smooths the noise away and can generalize worse — this is why batch size is tuned jointly with learning rate rather than picked independently.
Follow-up: "How does Adam differ from vanilla SGD, mechanically?"

Overfitting vs Underfitting

[FREQUENTLY ASKED]
  • Overfitting means the model memorizes training data noise and fails to generalize; underfitting means the model is too simple to capture the underlying pattern at all.
  • The bias-variance tradeoff exists because model error decomposes into bias (systematic error from oversimplifying) and variance (sensitivity to training data specifics) — you can't minimize both simultaneously without more data or better inductive bias.
  • The insight: regularization isn't "making the model dumber" — it's trading a small amount of bias for a large reduction in variance, which lowers total generalization error even though training loss gets worse.
RegimeTrain lossVal lossFix
Underfittinghighhighbigger model, more features, less regularization
Good fitlowlow—
Overfittinglowhighmore data, regularization, dropout, early stopping
▸ How this shows up in interviews
"Your validation loss is much higher than training loss. Walk me through how you'd debug it."
"I'd add dropout and more data."
First I'd confirm it's actually overfitting and not a data leak or distribution mismatch between train/val splits, since that produces the same symptom. Then I'd check the gap's severity relative to dataset size — with limited data, I'd add regularization (L2, dropout), early stopping, or data augmentation before reaching for a bigger dataset. I'd also sanity-check the split itself: temporal leakage, duplicate rows across train/val, or improperly correlated features are common causes that look like overfitting but aren't fixed by regularization at all.
Follow-up: "Why does L2 regularization reduce variance mechanically, not just empirically?"
