Supervised / Unsupervised / RL

[FREQUENTLY ASKED]
  • Supervised learning maps labeled inputs to outputs; unsupervised learning finds structure in unlabeled data; reinforcement learning learns a policy that maximizes cumulative reward through trial-and-error interaction with an environment.
  • These paradigms exist because the type of signal available (labels vs. no labels vs. delayed reward) constrains what algorithms are even possible.
  • The insight: RLHF sits at the intersection of all three — a reward model is trained supervised on human preference data, then used to fine-tune an LLM via RL — so "which paradigm" questions about modern LLMs often have a "more than one" answer.
▸ How this shows up in interviews
"How would you decide between supervised and unsupervised approaches for a new problem?"
"If I have labels I use supervised, otherwise unsupervised."
It starts with whether labels exist or are feasible to obtain — but the deeper question is what labels would cost versus what value they add. If labeling is expensive or the categories aren't known in advance (e.g., discovering customer segments), unsupervised methods like clustering surface structure you can act on or use to bootstrap weak labels. I'd also consider semi-supervised or self-supervised approaches as a middle ground — pretraining unsupervised on cheap unlabeled data, then fine-tuning supervised on a small labeled set, which is exactly the pattern behind most modern foundation models.
Follow-up: "Where does self-supervised learning fit in this taxonomy?"

Linear Regression

[FREQUENTLY ASKED]
  • Linear regression models a continuous target as a weighted sum of input features plus bias, fit by minimizing squared error between predictions and actual values.
  • It exists as the simplest possible model connecting inputs to a continuous output, giving a baseline every other regression method is measured against.
  • The insight: the normal equation gives an exact closed-form solution, but it's rarely used in practice because inverting X^TX is O(d³) in the number of features (plus O(nd²) just to form X^TX) and numerically unstable for collinear features — which is exactly why gradient descent is preferred at scale.
$$ \hat{y} = X\beta, \quad \mathcal{L} = \frac{1}{n}\sum_{i=1}^n (y_i - \hat{y}_i)^2, \quad \beta = (X^TX)^{-1}X^Ty $$
▸ How this shows up in interviews
"When would the normal equation fail or be a bad choice?"
"When there's too much data."
It fails numerically when X^TX is singular or near-singular — which happens with perfectly or highly collinear features — because the matrix inverse becomes unstable or undefined. It's also a bad choice computationally at scale: the O(d³) inversion cost in the feature dimension — plus the O(nd²) cost of forming X^TX over all samples — becomes intractable well before gradient-based methods would struggle, and gradient descent handles regularization (ridge/lasso) and streaming data naturally where the closed form doesn't.
Follow-up: "How does ridge regression fix the collinearity problem mathematically?"

Logistic Regression

[CRITICAL]
  • Logistic regression models the probability of a binary outcome by passing a linear combination of features through a sigmoid function, trained with cross-entropy loss.
  • It exists because linear regression's unbounded output is a poor fit for probabilities, which must lie in [0,1] — sigmoid squashes the linear score into that range.
  • The insight: cross-entropy loss isn't an arbitrary choice — it's the negative log-likelihood under a Bernoulli distribution, so "logistic regression" is really maximum likelihood estimation for a Bernoulli model, which is why the loss surface is convex and well-behaved unlike arbitrary loss choices.
$$ \sigma(z) = \frac{1}{1+e^{-z}}, \quad \mathcal{L} = -\frac{1}{n}\sum_i \left[ y_i \log\hat{y}_i + (1-y_i)\log(1-\hat{y}_i) \right] $$
▸ How this shows up in interviews
"Why not just use mean squared error for a classification problem?"
"Because cross-entropy is the standard for classification."
MSE combined with sigmoid produces a non-convex loss surface with flat regions where gradients vanish, since the sigmoid's derivative shrinks near 0 and 1 and MSE doesn't correct for that — training becomes slow and can get stuck. Cross-entropy's gradient with respect to the pre-sigmoid logit simplifies to (prediction − label), which stays large exactly when the model is confidently wrong, giving a much stronger and more stable training signal. It's also the theoretically correct loss under maximum likelihood for a Bernoulli-distributed target, not just an empirically convenient one.
Follow-up: "How would you handle severe class imbalance in a logistic regression setup?"

Decision Trees & Random Forests

[FREQUENTLY ASKED]
  • A decision tree splits data recursively on feature thresholds that maximize purity gain (Gini impurity or entropy reduction); a random forest averages many trees trained on bootstrapped samples with random feature subsets.
  • Ensembling exists because a single deep tree overfits badly (high variance) — averaging many decorrelated trees cancels out individual errors while preserving low bias.
  • The insight: random feature subsampling at each split isn't just for speed — it's what decorrelates the trees. Without it, all trees would find the same dominant feature first and the ensemble would barely reduce variance at all.

▸ How this shows up in interviews
"How does a random forest reduce variance compared to a single decision tree?"
"It uses more trees so it's more accurate."
Variance reduction from averaging only works if the individual trees' errors are decorrelated — random forests force that decorrelation via two mechanisms: bootstrap sampling (each tree sees a different data subset) and random feature selection at each split (each tree can't always split on the globally best feature). Mathematically, averaging N identically-distributed but correlated estimators reduces variance by a factor that depends on their pairwise correlation, not just N — so decorrelation is the actual lever, and just averaging more correlated trees quickly hits diminishing returns.
Follow-up: "Why does gradient boosting typically outperform random forests on tabular data, and what's the cost?"

Support Vector Machines

[ADVANCED]
  • SVMs find the hyperplane that maximizes the margin — the distance to the nearest points of each class (the support vectors) — rather than just any separating boundary.
  • Maximizing margin exists because it improves generalization: a boundary that barely separates classes is more likely to misclassify new points near the boundary than one with a wide buffer.
  • The insight: the kernel trick lets SVMs operate in an implicit, possibly infinite-dimensional feature space without ever computing coordinates in that space explicitly — you only ever need the kernel function (a similarity measure), not the transformed vectors themselves.
▸ How this shows up in interviews
"You have 10M labeled samples. Why is a kernel SVM a bad choice here, and what would you use instead?"
"SVMs are old — I'd just use a neural network."
Kernel SVMs need the pairwise kernel matrix, which is O(n²) in memory and worse in training time — at 10M samples that's structurally infeasible, not just slow, and prediction cost also grows with the number of support vectors, which itself grows with n. For tabular data at that scale I'd reach for gradient-boosted trees (near-SOTA on tabular, trains in minutes, handles mixed feature types), or a linear model with explicit feature crosses if latency matters most; a neural net only if the data has structure (images, text, sequences) that justifies it. The kernel trick itself is still worth understanding — it's the same "compute similarity without materializing the feature space" idea that shows up in attention approximations and kernel regression — but the algorithm's scaling wall is exactly why you rarely see SVMs in production anymore.
Follow-up: "Where does the kernel-trick idea reappear in modern deep learning?"

Evaluation Metrics

[CRITICAL]
  • Accuracy measures overall correctness; precision measures how many predicted positives are truly positive; recall measures how many actual positives were caught; F1 is their harmonic mean; AUC-ROC measures ranking quality across all thresholds.
  • Multiple metrics exist because "correct" means different things depending on the cost of false positives versus false negatives — a single number can't capture that asymmetry.
  • The insight: accuracy is actively misleading on imbalanced data — a model predicting the majority class every time scores high accuracy while being useless, which is exactly why fraud, medical, and rare-event models are evaluated on precision/recall/AUC instead.

▸ How this shows up in interviews
"You're building a cancer screening model. Which metric do you optimize for and why?"
"Accuracy, since we want the model to be right."
I'd prioritize recall, because a false negative (missed cancer) is far more costly than a false positive (an unnecessary follow-up test) — accuracy is actively dangerous here since the healthy class dominates the dataset. In practice I'd tune the classification threshold using a precision-recall curve to hit a target recall (e.g., 99%) and report the resulting precision, rather than picking a single blended metric like F1 that hides the asymmetry. I'd also watch for distribution shift — a model tuned on one population's disease prevalence can silently drift if deployed elsewhere.

Cross Validation

[FREQUENTLY ASKED]
  • K-fold cross validation splits data into K folds, trains on K-1 and validates on the remaining fold, rotating K times, then averages the results.
  • It exists because a single train/holdout split gives a noisy, sample-dependent performance estimate — averaging over K folds gives a lower-variance estimate of generalization.
  • The insight: cross-validation only estimates generalization correctly if the fold split respects the real deployment distribution — time-series data needs time-based splits (not random k-fold) or you leak future information into training, silently inflating your reported metrics.
▸ How this shows up in interviews
"Why is a single holdout split sometimes insufficient, and when is k-fold still the wrong tool?"
"K-fold is always better because it uses more data."
A single holdout split gives one noisy sample of performance that can look great or terrible purely from which rows landed in the split, especially with small datasets — k-fold averages over K such splits to reduce that variance. But standard k-fold assumes exchangeability across rows, which breaks for time-series (you'd train on future data to predict the past) or grouped data (same user's rows split across train and validation, leaking identity). In those cases you need time-based splits or group k-fold instead, or cross-validation gives you a confidently wrong number.
