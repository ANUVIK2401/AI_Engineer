01Beginner

What is RAG and Why — The Retrieval-Augmented Generation Pipeline

LLMs have a knowledge cutoff and don't know your private documents. RAG solves this by splitting the problem in two: a retrieval system fetches the most relevant context at query time, and the LLM generates an answer grounded in that context. The model reasons over real information instead of hallucinating from stale parametric memory.

Problem it solves

LLMs cannot access documents created after their training cutoff or any proprietary internal knowledge. RAG bridges this gap at query time without retraining.

Vs fine-tuning

Fine-tuning bakes knowledge into weights — but weights can't be updated on new data without retraining. RAG retrieves fresh content every query, so the knowledge base updates independently.

Grounding effect

When the LLM is given retrieved passages, it can cite them and is less likely to hallucinate. Real information crowds out confabulation because the model is biased toward what it sees in context.

💡
RAG ≠ just dumping documents into context The retrieval quality determines the answer quality. A RAG system with poor chunking, mismatched embedding models, or wrong retrieval strategy will produce worse answers than a well-prompted base LLM. The pipeline is only as good as its weakest step.
🔧
Why an AI Engineer needs this RAG is the default architecture for enterprise AI assistants, support bots, internal search, and document Q&A — the majority of production LLM systems. Understanding the full pipeline lets you diagnose whether a bad answer came from retrieval failure (wrong chunks returned), injection failure (too many/few chunks), or generation failure (LLM ignored the context).
02Intermediate

Vector Databases — Similarity Search & How to Choose

A vector database stores high-dimensional embeddings and supports approximate nearest-neighbour (ANN) search — finding the k vectors closest to a query vector in milliseconds across millions of documents. Under the hood, algorithms like HNSW build a graph-based index that trades a small recall hit for massive speed gains over brute-force search.

💡
ANN vs exact search Exact nearest-neighbour search is O(n·d) — at a million 1536-D vectors that's ~1.5 billion multiplications per query. HNSW finds the approximate top-k in O(log n) by navigating a multi-layer proximity graph. The recall trade-off (typically 95–99%) is almost always acceptable in production.
🔧
Which one to pick Start with Chroma locally. Move to pgvector if your team is already on Postgres and the corpus is under a few million vectors — you avoid a new service entirely. Choose Pinecone or Weaviate when you need managed scaling beyond a single Postgres instance, or when hybrid search (dense + BM25) is a first-class requirement.

Chunking Strategies — Fixed, Semantic & Hierarchical

How you split documents before embedding is one of the highest-leverage decisions in a RAG system. Too large → chunks contain multiple topics, embeddings average them, retrieval is imprecise. Too small → chunks lack context, the LLM can't understand what it retrieved. The right strategy depends on document structure.

04Intermediate

Embedding Models — How to Choose, OpenAI vs Open-Source

Every RAG system needs a text embedding model to map queries and documents into the same vector space. The choice affects retrieval quality, latency, cost, and whether you can run the model on-premise. The embedding model used at indexing time must match the one used at query time — mixing models silently destroys retrieval quality.

Decision: API vs on-premise

Use OpenAI embedding APIs when latency is non-critical, data privacy isn't a constraint, and you want zero infra. Use open-source (BGE-M3, E5) when data must stay on-premise, you have GPU capacity, or the corpus is multilingual.

Dimensionality trade-offs

Higher dimensions capture more semantic nuance but cost more storage and increase ANN query time. 1536D is the practical sweet spot for English. Matryoshka (MRL) training lets you truncate the vector at inference time — trade recall for speed without reindexing.

python 
from sentence_transformers import SentenceTransformer

# Local open-source embedding — no API key, no cost, no data leaves your machine
model = SentenceTransformer("BAAI/bge-m3")

docs = ["RAG improves factual grounding", "Vector databases store embeddings"]
vectors = model.encode(docs, normalize_embeddings=True)
# normalize_embeddings=True → cosine similarity == dot product (faster at query time)
💡
Critical rule — consistent model across index and query If you index with text-embedding-3-small and query with text-embedding-3-large, every cosine similarity score is meaningless. Retrieval returns random results. Track the embedding model name + version as metadata alongside each stored vector and validate it at application startup.

Hybrid Search — Dense + Sparse Retrieval

Dense retrieval (ANN on embeddings) excels at semantic matching — "car" retrieves "automobile". Sparse retrieval (BM25/TF-IDF) excels at exact keyword matching — "CVE-2024-1234" retrieves that exact string. Hybrid search fuses both signals, covering cases where either alone fails.

Reciprocal Rank Fusion (RRF)

The standard fusion formula: for each document, sum 1/(rank + 60) across all retrieval lists. The constant 60 prevents top-1 results from dominating. No score normalisation needed — only ranks matter. Simple, effective, no tuning required.

When to use hybrid

Always recommended for product search (SKUs, model numbers), technical docs (function names, error codes), medical/legal corpora (precise terminology). Dense-only is fine only when all content is general prose without identifiers.

python 
def reciprocal_rank_fusion(ranked_lists: list[list[str]], k=60) -> list[str]:
    scores: dict[str, float] = {}
    for ranked in ranked_lists:
        for rank, doc_id in enumerate(ranked, start=1):
            scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (rank + k)
    return sorted(scores, key=lambda d: -scores[d])

# dense_results and sparse_results are lists of doc_ids ordered by their score
fused = reciprocal_rank_fusion([dense_results, sparse_results])
06Advanced

Reranking — Why First-Pass Retrieval Isn't Enough

ANN retrieval is optimised for speed — it fetches top-100 candidates quickly but doesn't deeply model the query-document relationship. A cross-encoder reranker reads both query and document together, attending to every token interaction. It's slower but significantly more accurate, and it runs over a small candidate set so added latency is manageable.

Bi-encoder vs cross-encoder

Bi-encoder (embedding model): encodes query and doc independently → fast but less accurate. Use at retrieval stage to find top-100 candidates.

Cross-encoder (reranker): takes [query, doc] concatenated as one input, full attention between them → slow but highly accurate. Use over top-100 to select top-5.

Popular reranker models

Cohere Rerank API — managed, strong multilingual.
BGE-Reranker-v2 — open-source, strong English.
ms-marco-MiniLM-L-6-v2 — tiny, local, latency-friendly.
Jina Reranker v2 — long context (8k tokens), multilingual.

🔧
Two-stage retrieval pattern Retrieve top-100 with ANN (fast), rerank to top-5 with cross-encoder (accurate), inject top-5 into LLM context. The reranker adds ~50–150ms latency but typically improves answer quality more than any other single change you can make after basic RAG is working.
07Advanced

RAG Evaluation — Faithfulness, Relevance & Context Recall

A RAG system has two distinct failure modes: retrieval failure (wrong chunks returned) and generation failure (LLM ignored or contradicted the chunks). Evaluation frameworks like RAGAS separate these by measuring metrics that map cleanly onto each pipeline stage — so you know exactly what to fix.

RAGAS metric framework — what each metric measures in the pipeline
Faithfulness
0–1
Does the generated answer only contain claims inferable from retrieved context? Catches hallucinations — the LLM adding facts not in the chunks.
Answer Relevance
0–1
Is the answer actually relevant to the user's question? Catches verbose, off-topic, or incomplete answers even when the content is factually grounded.
Context Recall
0–1
What fraction of ground-truth answer claims are covered by retrieved context? Measures retrieval coverage — did we get back the right chunks?
Context Precision
0–1
Of the retrieved context, what fraction is relevant? High precision means retrieval isn't injecting noise that confuses the LLM.
Answer Correctness
0–1
End-to-end: is the answer factually correct relative to ground truth? Combines faithfulness + relevance into a single production KPI.
Noise Sensitivity
0–1
Does retrieval of irrelevant chunks degrade generation quality? Tests LLM robustness to context pollution from noisy retrieval.
python 
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_recall
from datasets import Dataset

data = {
    "question":     ["What is RAG?"],
    "answer":       ["RAG retrieves relevant documents before generating a response."],
    "contexts":     [["RAG stands for Retrieval-Augmented Generation. It fetches docs first."]],
    "ground_truth": ["RAG is a technique that augments LLM responses with retrieved context."],
}
result = evaluate(
    Dataset.from_dict(data),
    metrics=[faithfulness, answer_relevancy, context_recall]
)
print(result)
# {'faithfulness': 1.0, 'answer_relevancy': 0.93, 'context_recall': 0.88}

Diagnosing with metrics

Low context recall → retrieval problem. Fix: better chunking, more chunks, hybrid search, reranking.

Low faithfulness → generation problem. Fix: stronger prompt ("answer only from context"), larger top-k, better LLM.

Building an eval dataset

RAGAS can auto-generate question-answer pairs from your documents using an LLM. Start with 100–200 eval pairs covering diverse query types. Run eval on every pipeline change — chunk size, embedding model, top-k — to catch regressions before production.

🔧
Eval-driven RAG development Build the eval harness before optimising the pipeline. Without metrics, every "improvement" is a guess. The RAGAS loop: change one variable (chunk size, embedding model, top-k) → measure all metrics → keep the change only if context recall and faithfulness both improve. Never tune on the same split you evaluate on.
