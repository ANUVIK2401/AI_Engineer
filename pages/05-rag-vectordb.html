05 · RAG & Vector DBs

10 topics ~85 min deep-read Core [CRITICAL]

RAG is the most commonly deployed LLM architecture in production, so interviewers ask about it to check whether you understand the entire pipeline's failure modes, not just "it retrieves then generates."

RAG Pipeline End-to-End

[CRITICAL]
  • A user query is embedded into a vector, used to search a vector store for the most similar document chunks, which are then inserted into the LLM's prompt as context alongside the original query for generation.
  • RAG exists because LLMs have a fixed, frozen knowledge cutoff and finite parametric memory — retrieval lets you ground generation in current, private, or domain-specific information without retraining the model.
  • The insight: RAG doesn't make a model "know" anything — it changes what's in the context window at generation time. The model is still just predicting the next token, now conditioned on retrieved text; if retrieval fails, generation quality degrades exactly as if that context never existed.
▸ How this shows up in interviews
"Design a RAG system for a company's internal documentation. What are the key components?"
"I'd chunk the docs, embed them, store them in a vector DB, and retrieve at query time."
Beyond the basic pipeline, I'd specify the chunking strategy up front based on document structure (semantic or hierarchical chunking for structured docs rather than fixed-size), choose an embedding model validated against a benchmark relevant to the domain (not just the highest MTEB score), and add hybrid search combining dense retrieval with BM25 for exact term matches like product names or error codes. I'd also plan for evaluation from day one — a small labeled set of query/expected-answer pairs to measure retrieval recall and answer faithfulness — because without that, you can't tell if a prompt change or embedding model swap actually improved anything versus just changed behavior.
Follow-up: "How would you handle documents that get updated or deleted after being indexed?"

Chunking Strategies

[TRADEOFF]
  • Fixed-size chunking splits text by token/character count; semantic chunking splits at natural meaning boundaries (detected via embedding similarity shifts); hierarchical chunking preserves document structure (sections, subsections); sliding window chunking overlaps adjacent chunks to avoid cutting context at boundaries.
  • Chunking exists because embeddings and context windows both have size limits — you can't embed or retrieve an entire document as one unit, so it must be split into retrievable units small enough to be precise but large enough to retain context.
  • The insight: chunk size is a precision/recall tradeoff, not just a technical constraint — smaller chunks improve retrieval precision (less irrelevant text per chunk) but risk losing context needed to answer the question, while larger chunks preserve context but dilute the embedding's semantic focus and can bury the relevant sentence in irrelevant surrounding text.

▸ How this shows up in interviews
"Your RAG system retrieves technically relevant chunks but the LLM still gives incomplete answers. What's your first hypothesis?"
"I'd try a bigger embedding model."
My first hypothesis would be chunking, not embeddings — if the answer requires information that was split across two adjacent chunks (say, a definition in one chunk and its exception in the next), retrieval can return the technically most-similar chunk while missing the complementary information the model needed. I'd inspect the actual retrieved chunks against the ground-truth answer location in the source document, and if I find split answers I'd test adding overlap (sliding window) or switching to semantic/hierarchical chunking that keeps logically connected content together, rather than assuming the embedding model is the bottleneck.
Follow-up: "How does chunk overlap trade off against storage and retrieval cost?"

Embedding Models

[FREQUENTLY ASKED]
  • Embedding models are trained (often contrastively) to map semantically similar text to nearby vectors; quality is benchmarked on MTEB (Massive Text Embedding Benchmark) across retrieval, clustering, and classification tasks.
  • Open vs. API-based embedding models exist because teams trade off cost, latency, data privacy, and the ability to fine-tune the embedding model itself against the convenience of a managed API.
  • The insight: a model that tops the overall MTEB leaderboard isn't automatically the best choice — MTEB averages across many task types, so a model with a middling average score can still be the best specifically at retrieval (the sub-task RAG actually needs) — always check the task-specific score, not the aggregate.
▸ How this shows up in interviews
"How would you choose an embedding model for a RAG system?"
"I'd pick the top model on the MTEB leaderboard."
I'd look specifically at the retrieval sub-scores on MTEB, not the overall average, since RAG cares about retrieval quality specifically, not clustering or classification performance the aggregate score also folds in. Beyond the benchmark, I'd weigh practical constraints: embedding dimensionality (affects vector DB storage and search latency), whether the domain is specialized enough to need fine-tuning (medical/legal text often benefits from a domain-adapted model), and whether an API dependency is acceptable given latency and data privacy requirements. Ultimately I'd validate on a held-out set of real queries from the target domain, because public benchmarks don't always predict performance on your specific data distribution.
Follow-up: "What's the tradeoff of using a higher-dimensional embedding model?"

Vector Similarity Search

[FREQUENTLY ASKED]
  • Cosine similarity measures the angle between two vectors (direction only); dot product measures both angle and magnitude; L2 (Euclidean) distance measures straight-line distance between vector endpoints.
  • Multiple metrics exist because different embedding training objectives produce vectors where different notions of "similar" are meaningful — a model trained with cosine-based contrastive loss should be queried with cosine similarity, not L2.
  • The insight: using the wrong similarity metric for how a model was trained silently degrades retrieval quality without throwing any error — if an embedding model's training objective normalized vectors and optimized cosine similarity, querying with raw dot product or L2 distance can return a meaningfully different, worse-ranked result set.
MetricSensitive to magnitude?Use when
Cosine similaritynomodel trained with normalized/cosine objective
Dot productyesmodel trained with unnormalized dot-product objective
L2 distanceyesgeometric distance matters (e.g., clustering)
▸ How this shows up in interviews
"You swapped your vector DB's similarity metric from cosine to dot product and retrieval quality dropped. Why?"
"Dot product and cosine are basically the same thing."
They're only equivalent when vectors are normalized to unit length — dot product then reduces exactly to cosine similarity. If the embedding model wasn't trained with normalized outputs, raw dot product conflates vector magnitude with similarity, and magnitude often correlates with unrelated properties like text length or token frequency, not semantic relevance — so ranking by dot product on unnormalized vectors can surface longer or more "extreme" vectors regardless of true similarity. The fix is either to normalize vectors before indexing (making dot product and cosine equivalent) or to just match whichever metric the embedding model's training objective actually optimized for.
Follow-up: "Why might dot product actually be preferred over cosine for certain retrieval tasks despite this risk?"

Vector DBs Comparison

[TRADEOFF]
  • Pinecone, Weaviate, Chroma, pgvector, and Qdrant are all systems for storing embeddings and performing approximate nearest-neighbor search, differing in deployment model, scale, filtering capability, and operational overhead.
  • Multiple vector DBs exist because teams have different constraints: some need a fully-managed service with minimal ops burden, others need to keep data in an existing Postgres instance, others need massive scale with fine-grained metadata filtering.
  • The insight: pgvector isn't a "toy" option — for datasets under a few million vectors, running vector search inside your existing Postgres instance avoids an entirely new piece of infrastructure to operate, which is often the right tradeoff over a "purpose-built" vector DB that adds operational surface area for marginal performance gains at that scale.

▸ How this shows up in interviews
"When would you choose pgvector over a purpose-built vector database like Pinecone?"
"pgvector is free and Pinecone costs money."
If the team already operates Postgres and the dataset is in the low millions of vectors, pgvector avoids introducing an entirely new system to deploy, monitor, and secure — you get transactional consistency with your existing relational data, and joins between vector search results and relational metadata are trivial. The tradeoff shows up at larger scale or with very high query throughput requirements, where purpose-built vector DBs have more mature ANN index implementations and horizontal scaling built in — pgvector's HNSW support has improved a lot but still has real limits before index build time and query latency become bottlenecks that justify migrating.

HNSW Indexing

[ADVANCED]
  • HNSW (Hierarchical Navigable Small World) builds a multi-layer graph of vectors where higher layers have fewer, more widely-connected nodes for fast coarse navigation, and lower layers have dense connections for precise local search.
  • Approximate nearest neighbor (ANN) search exists because exact nearest-neighbor search requires comparing a query against every vector in the dataset — linear in dataset size, which becomes too slow at millions/billions of vectors.
  • The insight: HNSW trades a small amount of recall (you might miss the true nearest neighbor occasionally) for logarithmic-ish search time — this is a deliberate, tunable tradeoff (via parameters like ef_search), not an unavoidable flaw, and most production RAG systems never notice the approximation because top-k retrieval is already forgiving of small ranking errors.
▸ How this shows up in interviews
"Why do we need approximate nearest neighbor search instead of just computing exact distances?"
"It's faster."
Exact nearest-neighbor search requires computing the distance from the query to every single vector in the index, which is O(n) per query — at a billion vectors that's computationally infeasible for real-time retrieval latency requirements. HNSW instead builds a navigable graph structure where search starts at a coarse, sparse top layer and progressively narrows into denser, more precise lower layers, achieving roughly logarithmic search complexity at the cost of occasionally missing the exact true nearest neighbor. In practice this approximation is tunable (via ef_search / ef_construction parameters) and the recall loss is negligible for top-k retrieval used in RAG, where being in the top 5 most relevant chunks matters more than being ranked exactly first.
Follow-up: "What's the tradeoff between HNSW's ef_construction parameter and index build time?"

Hybrid Search

[CRITICAL]
  • Hybrid search combines dense vector retrieval (semantic similarity) with sparse retrieval like BM25 (exact term/keyword matching), then merges the two ranked lists (often via reciprocal rank fusion).
  • It exists because dense embeddings, trained on general semantic similarity, systematically underperform on queries containing rare terms, product codes, acronyms, or exact identifiers that never carried strong semantic signal during training.
  • The insight: pure dense retrieval failing on rare terms isn't a bug you can fix by picking a better embedding model — it's structural, because embeddings compress meaning and necessarily lose exact lexical identity; you need a fundamentally different retrieval mechanism (sparse/lexical) to recover it, not a better version of the same mechanism.

▸ How this shows up in interviews
"Your RAG system fails on queries containing specific error codes, but works fine on conceptual questions. What's the fix?"
"I'd fine-tune the embedding model on more examples."
This is a textbook signal for adding hybrid search — dense embeddings compress text into a semantic space that's excellent at capturing conceptual similarity but structurally weak at exact-match tasks like an alphanumeric error code, since the training objective never rewarded preserving exact token identity. I'd add a sparse retrieval path (BM25) running in parallel with dense retrieval, then merge results via reciprocal rank fusion or a learned reranker, so the exact-match strength of BM25 covers the failure case dense search structurally can't. Fine-tuning the embedding model could help marginally, but it's treating a structural retrieval-method gap as a training data problem, which is a slower and less reliable fix.
Follow-up: "How does reciprocal rank fusion combine two differently-scaled ranking lists?"

Reranking

[FREQUENTLY ASKED]
  • A bi-encoder embeds query and documents separately and compares vectors, enabling fast precomputed search; a cross-encoder jointly processes the query and a candidate document together through a full transformer, producing a much more accurate but far slower relevance score.
  • Reranking exists to combine both strengths: use a fast bi-encoder to retrieve a broad candidate set (e.g., top 100), then apply a slow but accurate cross-encoder to rerank just that shortlist down to the final top-k shown to the LLM.
  • The insight: a cross-encoder can't be used for the initial retrieval step at all — it requires the query at scoring time, so document representations can't be precomputed and indexed, making it computationally impossible to run at index-wide scale. Reranking is the compromise that captures its accuracy benefit only where it's affordable.
▸ How this shows up in interviews
"Why not just use a cross-encoder for retrieval directly and skip the bi-encoder step?"
"Cross-encoders are too slow."
It's more fundamental than speed — a cross-encoder needs both the query and the candidate document as joint input to compute a score, meaning you cannot precompute document representations ahead of time the way you can with a bi-encoder's independent embeddings. That means using a cross-encoder for full-index retrieval would require running a full transformer forward pass for every single document in the index, for every query, which is computationally infeasible at any meaningful scale. The two-stage pipeline — bi-encoder for cheap, precomputed, approximate retrieval over the whole index, cross-encoder only on the small shortlist — is the only way to get the cross-encoder's accuracy without paying its full cost.
Follow-up: "How many candidates would you typically retrieve before reranking, and what determines that number?"

RAG Failure Modes

[CRITICAL]
  • RAG can fail via hallucination despite correct retrieval (the model ignores or misreads context), retrieval misses (the right chunk was never surfaced), or context overflow (too many retrieved chunks exceed the context window or dilute attention over irrelevant text).
  • These failure modes exist because RAG has multiple independent stages (embedding, search, ranking, generation), each with its own failure surface — a single "it hallucinated" symptom can trace back to any of them.
  • The insight: most teams debug RAG failures by tweaking the prompt first, but the majority of RAG quality issues are retrieval failures, not generation failures — you have to isolate which stage failed (by manually inspecting what was actually retrieved) before you know which lever to pull.

▸ How this shows up in interviews
"Your RAG chatbot gave a wrong answer. How do you debug it systematically?"
"I'd look at the prompt and try to improve it."
I'd start by isolating which pipeline stage actually failed rather than guessing — first checking what chunks were literally retrieved for that query, since if the correct information was never retrieved, no prompt engineering fixes it. If retrieval did surface the right chunk, I'd check whether it's a "lost in the middle" issue from too much or poorly-ordered context, versus a genuine faithfulness failure where the model had the right information but ignored or contradicted it. Each of those has a different fix — retrieval misses point to chunking/embedding/hybrid search issues, context overload points to reducing retrieved chunk count or reordering, and faithfulness failures point to prompt instructions or a different generation model — so diagnosis has to come before any fix.
Follow-up: "What is the 'lost in the middle' phenomenon and why does it happen even in models with long context windows?"

RAG Evaluation (RAGAS)

[CRITICAL]
  • RAGAS-style evaluation measures faithfulness (is the generated answer supported by the retrieved context), answer relevance (does the answer address the actual question), and context recall (did retrieval surface the necessary information).
  • These specific metrics exist because "the answer is wrong" is too coarse a signal to act on — you need to know whether the failure is in retrieval or generation, and faithfulness/relevance/recall decompose the pipeline into independently measurable, actionable components.
  • The insight: faithfulness and correctness are different things — a response can be perfectly faithful to retrieved context (never contradicting it) while the retrieved context itself was wrong or incomplete, so a high faithfulness score doesn't guarantee a factually correct final answer; you need context recall/precision evaluated separately.

▸ How this shows up in interviews
"How would you set up automated evaluation for a RAG system before it goes to production?"
"I'd manually test it with some sample questions."
I'd build a labeled evaluation set of representative queries paired with ground-truth answers and the source documents that should be retrieved, then compute retrieval-stage metrics (context recall/precision) separately from generation-stage metrics (faithfulness, answer relevance) using a framework like RAGAS, often with an LLM-as-judge for the generation metrics since they require semantic judgment rather than exact match. Separating the stages is the key design decision — it means when overall accuracy drops after a change, I can immediately tell whether the regression came from retrieval or generation instead of re-debugging the whole pipeline from scratch. I'd also track this over time as a regression suite, not just a one-time check before launch, since embedding model updates or prompt changes can silently regress either stage.
Follow-up: "What are the risks of using an LLM-as-judge to evaluate faithfulness, and how would you mitigate them?"
