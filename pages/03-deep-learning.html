Neural Net Forward Pass

[FREQUENTLY ASKED]
  • A forward pass computes layer outputs sequentially: each layer applies a linear transform (weights + bias) followed by a nonlinear activation function, feeding into the next layer.
  • Stacking layers exists because a single linear layer can only represent linear functions — nonlinear activations between layers are what let deep networks approximate arbitrary functions.
  • The insight: without a nonlinearity between layers, stacking any number of linear layers collapses mathematically into a single linear layer — depth without nonlinearity buys you nothing.
▸ How this shows up in interviews
"Why do we need nonlinear activation functions between layers?"
"To make the network more powerful."
Composing linear transformations is still linear — W2(W1x + b1) + b2 simplifies to a single effective weight matrix and bias, so a 100-layer network without nonlinearities has the exact same representational capacity as one layer. Nonlinear activations break that collapse, letting each layer carve the input space in a genuinely new way, which is what allows deep networks to approximate complex, non-linear decision boundaries or functions. This is also why activation choice matters for trainability, not just expressivity — some (sigmoid) saturate and kill gradients, others (ReLU) don't.
Follow-up: "What would happen if you used a linear activation function throughout an entire deep network?"

Backpropagation & Chain Rule

[CRITICAL]
  • Backprop computes the gradient of the loss with respect to every parameter by applying the chain rule backward through the computation graph, layer by layer.
  • It exists because computing each parameter's gradient independently from scratch would be exponentially expensive — backprop reuses intermediate results, making gradient computation roughly as cheap as the forward pass.
  • The insight: backprop is just reverse-mode automatic differentiation applied to a specific graph shape — there's no separate "learning algorithm," it's mechanical calculus plus dynamic programming to avoid redundant computation.
$$ \frac{\partial \mathcal{L}}{\partial w^{(1)}} = \frac{\partial \mathcal{L}}{\partial a^{(L)}} \cdot \frac{\partial a^{(L)}}{\partial a^{(L-1)}} \cdots \frac{\partial a^{(2)}}{\partial w^{(1)}} $$
▸ How this shows up in interviews
"Explain backprop to someone who understands calculus but not deep learning."
"It sends the error backward through the network to update weights."
Backprop is the chain rule applied systematically: to know how much the loss changes if you nudge an early-layer weight, you multiply the local derivatives along the path from that weight to the final loss, layer by layer, working backward. The reason this is efficient rather than exponential is that we cache each layer's local gradient once and reuse it for every weight that feeds into it, rather than recomputing the whole chain from scratch per parameter — this reuse is the actual algorithmic insight, not the calculus itself.
Follow-up: "Why does the vanishing gradient problem get worse with more layers, mechanically?"

Activation Functions

[TRADEOFF]
  • ReLU, GELU, sigmoid, and tanh are nonlinear functions applied after each linear layer, each with different gradient behavior and saturation properties.
  • Multiple activations exist because each makes a different tradeoff between gradient flow, computational cost, and output range needs.
  • The insight: ReLU's "dying neuron" problem (gradient is exactly zero for negative inputs) is a feature as much as a bug — it induces sparsity — but GELU's smooth, probabilistic gating avoids the hard cutoff entirely, which is why transformers (BERT, GPT) use GELU instead of ReLU.

▸ How this shows up in interviews
"Why did transformers move from ReLU to GELU?"
"GELU just performs better empirically."
ReLU has a hard, non-differentiable kink at zero and completely zeroes out negative inputs, which can cause "dead neurons" that never recover since their gradient is permanently zero. GELU weights inputs by their value under the standard normal CDF, giving a smooth, differentiable curve that still allows some signal through small negative inputs — this smoother gradient landscape empirically helps optimization in very deep, wide architectures like transformers. It's a small architectural choice, but at transformer scale small consistent gradient improvements compound significantly over training.
Follow-up: "What's the 'dying ReLU' problem and how does Leaky ReLU address it?"

Batch Norm vs Layer Norm

[CRITICAL]
  • Batch norm normalizes activations across the batch dimension per feature; layer norm normalizes across the feature dimension per individual sample.
  • Normalization exists to stabilize the distribution of activations flowing through the network, letting you use higher learning rates — historically attributed to reducing "internal covariate shift," though later work (Santurkar et al., 2018) showed the real mechanism is smoothing the loss landscape. Citing that correction is itself a senior-signal in interviews.
  • The insight: batch norm's statistics depend on other examples in the batch, which breaks down for variable-length sequences and small/streaming batches (like autoregressive generation) — that's precisely why transformers use layer norm, which normalizes independently per token regardless of batch size or sequence position.

▸ How this shows up in interviews
"Why do transformers use layer norm instead of batch norm?"
"Because it works better for sequences."
Batch norm computes mean and variance across the batch dimension for each feature, which requires a reasonably large, consistent batch to get stable statistics — but during autoregressive generation you often decode one token at a time with batch size 1, where batch statistics are meaningless. Layer norm instead normalizes across the feature dimension within a single token's representation, so it works identically regardless of batch size or sequence length, which is exactly the flexibility transformer training and inference need. There's also a variable-length sequence problem: padded tokens would corrupt batch statistics in batch norm, whereas layer norm sidesteps this entirely.
Follow-up: "What is RMSNorm and why do some modern LLMs (LLaMA) use it instead of standard LayerNorm?"

Dropout

[TRADEOFF]
  • Dropout randomly zeroes a fraction of neurons during each training step, forcing the network to not rely on any single neuron or co-adapted group of neurons.
  • It exists as a cheap regularization technique that approximates training an ensemble of exponentially many thinned sub-networks and averaging them.
  • The insight — the classic interview gotcha: dropout must be disabled at inference time, and the remaining activations must be rescaled (inverted dropout divides by keep-probability during training) so that expected activation magnitude matches between train and inference — forgetting this silently degrades model output without throwing any error.
▸ How this shows up in interviews
"What happens if you forget to disable dropout at inference time?"
"The model would still work but be a bit less accurate."
You'd get nondeterministic outputs for the same input — because dropout randomly zeroes different neurons each forward pass — which is immediately disqualifying for any production system expecting reproducible predictions. Beyond nondeterminism, the network was trained expecting a certain fraction of neurons active, and randomly zeroing more of them at inference systematically shifts the activation distribution the downstream layers were calibrated for, degrading accuracy in a way that won't throw an error, just silently worse predictions. This is a real production bug class — frameworks separate train()/eval() modes specifically to prevent it, and forgetting to call model.eval() is a common and hard-to-notice mistake.
Follow-up: "How is inverted dropout different from vanilla dropout, and why does it matter for inference?"

Convolutional Neural Networks

[FREQUENTLY ASKED]
  • A convolution slides a small learnable filter across the input, computing local dot products to detect spatial patterns like edges or textures; pooling downsamples to reduce spatial resolution and add translation invariance.
  • Parameter sharing (the same filter applied everywhere) exists because a pattern like "vertical edge" is useful regardless of where it appears in the image — reusing weights drastically cuts parameter count versus a fully connected layer.
  • The insight: CNNs encode a strong inductive bias — locality and translation invariance — which is exactly right for images but exactly wrong for tasks without spatial locality, which is part of why attention-based architectures (with no such bias) generalize better across modalities.
▸ How this shows up in interviews
"Why does parameter sharing in convolutions matter beyond just saving memory?"
"It reduces the number of weights so training is faster."
Parameter sharing encodes the assumption that a feature detector (like an edge or texture detector) is useful regardless of its position in the image, so the same filter weights are reused at every spatial location instead of learning position-specific weights. This is what gives CNNs translation invariance — a cat detected in the top-left uses the same learned filter as a cat in the bottom-right — and it also means the effective sample efficiency is much higher, since every patch of every image contributes to training the same shared filter rather than one image only training weights for its exact position.
Follow-up: "Why do CNNs eventually lose to vision transformers on very large datasets?"

RNNs / LSTMs

[FREQUENTLY ASKED]
  • RNNs process sequences step by step, carrying a hidden state forward; LSTMs add gating mechanisms (forget, input, output gates) to control what information persists in that hidden state over time.
  • Gating exists because vanilla RNNs suffer from vanishing/exploding gradients across long sequences — repeatedly multiplying the same recurrent weight matrix during backprop-through-time shrinks or blows up gradients exponentially with sequence length.
  • The insight: transformers didn't replace RNNs because attention is inherently "smarter" — they replaced RNNs because sequential recurrence can't be parallelized across time steps during training, while attention computes all pairwise interactions simultaneously, making transformers dramatically more efficient to train at scale on modern hardware.

▸ How this shows up in interviews
"Why exactly did transformers replace RNNs, mechanistically?"
"Transformers are just better at understanding language."
The core bottleneck wasn't modeling quality alone — it was that RNNs process tokens sequentially, so computing the hidden state at position t requires the hidden state at t-1, making training inherently non-parallelizable across the sequence dimension. Transformers replace recurrence with self-attention, which computes relationships between all token pairs simultaneously via matrix multiplication, so the entire sequence can be processed in parallel on a GPU. This unlocked training on far larger datasets within practical time budgets, which is arguably the single largest reason LLMs became possible — not that attention understands language better in principle, but that it scales.
Follow-up: "If attention scales so well, why do some architectures still combine it with recurrence (e.g., in state-space models)?"

Transfer Learning

[FREQUENTLY ASKED]
  • Transfer learning reuses a model pretrained on a large, general dataset and adapts it to a new, usually smaller and more specific task, either by fine-tuning some/all layers or using it as a frozen feature extractor.
  • It exists because training from scratch requires enormous labeled data and compute that most tasks don't have — pretrained representations already encode broadly useful features (edges, syntax, semantics) that transfer across tasks.
  • The insight: which layers to freeze depends on how similar the new task is to the pretraining task — early layers learn generic, broadly transferable features (edges, low-level syntax) while later layers specialize toward the original task, so the more dissimilar your target task, the more layers you should unfreeze and retrain.
▸ How this shows up in interviews
"You have a small labeled dataset for a niche image classification task. How do you decide what to freeze?"
"I'd freeze most layers since I don't have much data."
The decision hinges on two axes: dataset size and similarity to the pretraining domain. With a small dataset and a task similar to the pretraining data, I'd freeze most of the backbone and only fine-tune the final classifier layer to avoid overfitting on limited examples. If the task is dissimilar to the pretraining domain (e.g., medical imaging vs. ImageNet's natural photos) even with limited data, I'd unfreeze more of the later layers since the pretrained features may not transfer well, accepting some overfitting risk mitigated by a low learning rate and strong regularization — freezing everything in that case would just underfit.
Follow-up: "Why do you typically use a much lower learning rate when fine-tuning versus training from scratch?"
