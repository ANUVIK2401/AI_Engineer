08 · MLOps & Production

9 topics ~70 min deep-read Advanced [FREQUENTLY ASKED]

Interviewers ask about this because a model that works in a notebook and a model that survives production for a year are entirely different engineering problems — this module is where "can you ship and maintain this" gets tested.

ML Lifecycle

[FREQUENTLY ASKED]
  • The ML lifecycle runs data collection/preparation → training → evaluation → deployment → monitoring, then feeds monitoring signals back into retraining — it's a loop, not a one-way pipeline.
  • It exists as a structured process because each stage has distinct failure modes and tooling needs, and skipping the loop-back (monitoring informing retraining) is exactly how models silently degrade in production without anyone noticing.
  • The insight: most of the actual engineering effort in a mature ML system goes into the "boring" stages — data pipelines, monitoring, retraining automation — not model architecture; teams that only invest in the training/eval steps end up with models that work once and degrade silently afterward.
▸ How this shows up in interviews
"A model performed great in evaluation but degrades after a few months in production. Walk me through why this happens and how you'd design against it."
"I'd retrain the model periodically."
The root cause is almost always that the offline evaluation captured a snapshot of the world that has since shifted — either the input data distribution changed (data drift) or the underlying relationship between inputs and the target changed (concept drift) — and without production monitoring, this degradation is invisible until someone notices business metrics slipping. I'd design the lifecycle as a closed loop from the start: instrument production predictions and (where available) eventual ground truth outcomes, track distributional statistics of incoming features against the training distribution, and set concrete alerting thresholds that trigger investigation or automated retraining rather than relying on a fixed retraining calendar that might be too slow or wasteful depending on how fast the underlying data actually shifts.
Follow-up: "What's the difference between scheduled retraining and triggered retraining, and when would you use each?"

Model Versioning

[TRADEOFF]
  • Model versioning tracks not just model weights but the training data snapshot, hyperparameters, code version, and evaluation metrics associated with each trained artifact.
  • It's harder than software versioning because model behavior depends on data and randomness in training, not just code — two runs of "the same" training script can produce meaningfully different models, and reproducing a specific model requires pinning data, code, and often random seeds together as one unit.
  • The insight: a model version without its associated training data snapshot and eval results is nearly useless for debugging — if a production issue traces back to a specific model version, you need to know exactly what data and code produced it to understand and fix the root cause, not just roll back to a different weights file.
▸ How this shows up in interviews
"Why is versioning a model harder than versioning a regular software artifact?"
"Because models are bigger files."
Software versioning is fundamentally about code, which is deterministic — the same commit always produces the same binary. Model versioning has to account for the training data snapshot (which can change independently of code), hyperparameters, and often non-determinism from random initialization or data shuffling, meaning the same code and even the same data can produce meaningfully different model weights across runs. A complete model version therefore needs to bundle the weights with the exact data snapshot, code version, hyperparameters, and evaluation results as one traceable unit — without that, if a production issue surfaces, you can't reliably reproduce or root-cause which factor (data change vs. code change vs. random variation) actually caused the behavior difference.
Follow-up: "How would you version a fine-tuned LLM when the base model itself also gets updated periodically?"

Feature Stores

[ADVANCED]
  • A feature store is a centralized system for computing, storing, and serving features consistently for both training (offline, batch) and inference (online, low-latency) use cases.
  • It exists to solve training/serving skew — the problem where features are computed differently (different code paths, different timing) between the offline training pipeline and the online serving path, causing subtle, hard-to-detect prediction quality issues.
  • The insight: training/serving skew is one of the most common and hardest-to-detect production ML bugs precisely because it doesn't throw an error — the model just silently performs worse than its offline evaluation suggested, because the online features it's actually receiving are computed slightly differently than the ones it was trained and evaluated on.
▸ How this shows up in interviews
"What specific problem does a feature store solve that you couldn't solve with good documentation and code review alone?"
"It keeps features organized and easy to find."
The core problem is training/serving skew — even with perfect documentation, if your training pipeline computes a feature via a batch SQL job and your serving path computes the "same" feature via a different real-time code path, subtle differences (timing windows, null-handling, rounding) will inevitably creep in over time as each path is maintained independently by potentially different people. A feature store enforces a single feature definition used by both the offline training pipeline and the online serving path, so the exact same computation logic produces both the training data and the inference-time features, structurally eliminating the skew rather than relying on developers remembering to keep two implementations in sync. This is a systems/architecture fix for a class of bug that code review alone reliably fails to catch, since the two code paths are often in entirely different services.
Follow-up: "How does a feature store handle a feature that requires real-time computation, like 'transactions in the last 5 minutes'?"

Data Drift vs Concept Drift vs Model Decay

[CRITICAL]
  • Data drift is a shift in the input feature distribution while the input-output relationship stays the same; concept drift is a shift in the input-output relationship itself; model decay is the resulting degradation in model performance from either (or from staleness generally).
  • These distinctions exist because they require different detection methods and different fixes — data drift can sometimes be detected without labels (comparing feature distributions), while concept drift requires tracking actual prediction accuracy against ground truth over time.
  • The insight: data drift can be detected without waiting for ground-truth labels (by monitoring input feature distributions directly), but concept drift often can't be detected until labels arrive — which means for tasks with delayed labels (like loan default prediction, resolved months later), concept drift can be silently accumulating for a long time before you have any direct evidence of it.
TypeWhat changesDetectable without labels?
Data driftinput feature distributionyes — compare feature stats over time
Concept driftinput→output relationshipno — needs ground truth labels
Model decayoverall performance (either cause)only if labels or proxy metrics available
▸ How this shows up in interviews
"How would you detect concept drift in a system where ground-truth labels arrive 3 months after prediction?"
"I'd just wait for the labels and check accuracy."
Waiting passively for delayed labels means you're always 3 months behind on detecting an actual problem, which is too slow for many production risk profiles — instead I'd build proxy signals that correlate with concept drift without needing the true label: monitoring input feature distributions for drift (a leading indicator, even though it's technically a different phenomenon), tracking the model's prediction distribution itself for unexpected shifts, and where possible, using a faster-arriving weak label or human-reviewed sample to get an early, noisier signal on accuracy well before the full 3-month ground truth resolves. I'd treat the 3-month labels as the authoritative signal for triggering retraining decisions, but use the faster proxy signals as an early-warning system to flag investigation before waiting for full confirmation.

A/B Testing for Models

[FREQUENTLY ASKED]
  • Shadow mode runs a new model alongside production without serving its predictions to users, comparing outputs offline; canary deployment routes a small percentage of real traffic to the new model; full A/B testing splits traffic between model versions and measures the business metric difference statistically.
  • This staged rollout process exists because offline evaluation metrics don't always predict real-world business impact, and deploying untested changes directly to 100% of traffic risks a broad, immediate negative impact if something was missed in evaluation.
  • The insight: shadow mode catches technical/operational failures (crashes, latency, wildly wrong outputs) safely with zero user risk, but it can't measure actual business impact, since shadow predictions never affect real user behavior — you need a real canary or A/B test specifically to validate the metric that actually matters, and skipping straight from shadow mode to full rollout skips that validation entirely.
▸ How this shows up in interviews
"Your new model passed shadow mode with no errors. Is that enough to roll out to 100% of traffic?"
"Yes, if it ran without errors in shadow mode it's safe to deploy fully."
Shadow mode only validates that the new model runs correctly and produces outputs in a similar range to the old model — it never actually influences real user behavior, so it can't tell you anything about the metric you actually care about, like conversion rate or user satisfaction. I'd follow shadow mode with a canary deployment to a small percentage of real traffic, measuring the actual business metric with statistical rigor (accounting for sample size and variance) before considering a full rollout, since real user response to model changes can differ in ways that never show up when the model's predictions have zero real-world consequence. Skipping from shadow mode straight to 100% rollout conflates "doesn't crash" with "actually improves the metric," which are very different bars.
Follow-up: "How would you determine the right traffic percentage and duration for a canary deployment?"

Containerization for ML

[FREQUENTLY ASKED]
  • An ML serving container typically bundles the model weights, inference code, exact library versions (especially deep learning framework and CUDA versions), and a runtime environment, packaged to run identically regardless of the host machine.
  • Containerization exists because ML dependency environments are notoriously fragile — CUDA version, framework version, and even specific driver versions can silently change numerical behavior or break compatibility, and containers pin the entire environment as one reproducible unit.
  • The insight: for ML specifically, "it works on my machine" is a much deeper problem than in typical software — GPU driver/CUDA version mismatches can cause a model to run without any error but produce subtly different numerical outputs, which is a much scarier failure mode than a container that simply fails to start.
▸ How this shows up in interviews
"Why is 'it works on my machine' a more dangerous problem for ML systems than for typical software?"
"Because ML has more dependencies to manage."
In typical software, a dependency mismatch usually causes an obvious crash or import error — annoying but easy to detect. In ML specifically, a CUDA version mismatch, a different BLAS backend, or even a different GPU architecture can cause the model to run successfully and produce numerically slightly different outputs without any error at all, since floating-point operations aren't perfectly deterministic across all hardware/library combinations. That means a silently different environment can degrade model quality in production in a way that's invisible until someone notices worse business metrics, which is exactly why pinning the full environment (framework version, CUDA version, even specific driver compatibility) inside a container matters more for ML than for typical stateless web services.

Observability

[CRITICAL]
  • ML observability logs inputs, outputs, prediction confidence/latency, token counts (for LLMs), and errors, in addition to standard system metrics like CPU/memory/request rate.
  • This exists because standard software monitoring (uptime, latency, error rate) tells you the system is running, but says nothing about whether the model's predictions are still good — you need model-specific signals layered on top of infra monitoring.
  • The insight: a model can have perfect infrastructure health (100% uptime, low latency, zero 500 errors) while silently producing garbage predictions — infra monitoring and model quality monitoring are almost entirely orthogonal, and teams that only instrument the former have no visibility into the failure mode that actually matters most.
▸ How this shows up in interviews
"Your dashboards show 100% uptime and low latency, but users are complaining the model's outputs are bad. What's missing from your monitoring?"
"We should add more logging."
What's missing is model-quality-specific observability layered on top of infrastructure monitoring — uptime and latency confirm the service is running, but say nothing about whether predictions are accurate, well-calibrated, or even reasonable, since a model can serve confidently wrong outputs at perfect infra health. I'd instrument prediction distributions (are outputs drifting from historical patterns), confidence/uncertainty scores where available, and for LLMs specifically, things like token usage, refusal rates, and sampled output review — plus, where feasible, a feedback loop capturing implicit or explicit user signals (thumbs down, retry rate, session abandonment) as a proxy for quality when ground truth labels aren't immediately available. This is the layer most teams under-invest in because it's harder to instrument than standard infra metrics, but it's the layer that actually explains user-reported quality complaints.

CI/CD for ML

[TRADEOFF]
  • ML CI/CD extends standard software CI/CD (test, build, deploy) with model-specific stages: data validation, model evaluation against a held-out benchmark, and often a staged rollout (shadow/canary) rather than instant full deployment.
  • It differs from software CI/CD because "does this pass unit tests" isn't sufficient — a model can pass all code tests while still being a worse model than the one it's replacing, so an explicit evaluation gate comparing against the current production model's metrics is required before promoting.
  • The insight: in software CI/CD, passing tests is usually a reliable signal to deploy; in ML CI/CD, passing tests only proves the code runs — the actual "does this improve things" question requires a separate evaluation gate against real metrics, meaning ML CI/CD pipelines need an additional decision point that pure software pipelines don't.
▸ How this shows up in interviews
"How is a CI/CD pipeline for an ML model fundamentally different from one for a typical microservice?"
"It just has an extra step to train the model."
A microservice's CI/CD pipeline treats passing tests as sufficient evidence to deploy, because code behavior is deterministic given the same inputs — if tests pass, the service does what it's supposed to. An ML pipeline can't rely on that same logic: code can pass every unit test (the training script runs, the API returns valid JSON) while producing a model that's measurably worse than the current production model on the metric that actually matters. This means ML CI/CD needs an explicit evaluation gate — running the newly trained model against a fixed benchmark and comparing to the current production model's performance — as a distinct decision point before promotion, which most standard software CI/CD pipelines have no equivalent for. Staged rollout (shadow/canary) is also far more standard in ML pipelines specifically because offline evaluation doesn't always predict production impact reliably.
Follow-up: "What would you do if a newly trained model passes the evaluation gate but a stakeholder is still uneasy about deploying it?"

Cloud AI Services

[TRADEOFF]
  • SageMaker (AWS), Vertex AI (Google Cloud), and Azure ML are managed platforms bundling training infrastructure, model registries, deployment endpoints, and monitoring into one ecosystem, reducing the operational burden of building these systems from scratch.
  • They exist because building and maintaining the full MLOps stack (feature stores, training orchestration, serving infra, monitoring) in-house is a significant, ongoing engineering investment that most teams don't need to reinvent when a managed option covers their needs.
  • The insight: the right time to reach for a managed platform is a team-size and workload-maturity decision, not a technical superiority question — small teams benefit enormously from offloading undifferentiated MLOps infrastructure work, but teams with very specific performance requirements or existing heavy investment in open-source tooling (vLLM, MLflow, Kubernetes) can find managed platforms constraining rather than helpful past a certain scale.

▸ How this shows up in interviews
"When would you NOT recommend using a managed platform like SageMaker or Vertex AI?"
"When it's too expensive."
Cost is one factor, but the more fundamental case is when your requirements are specific enough that the managed platform's abstractions get in the way rather than help — for example, if you need a specific serving optimization like vLLM's paged attention that isn't well-supported inside the platform's serving abstraction, or your team already has significant investment and expertise in open-source tooling (Kubernetes, MLflow, Ray) that a managed platform would partially duplicate or conflict with. I'd also be cautious recommending a managed platform for a team planning to scale into very high-volume, cost-sensitive serving, since the convenience premium on managed compute can become a significant line item at scale compared to a well-run in-house or open-source-based stack — the right call depends on team size, in-house MLOps maturity, and how standard versus specialized the workload is.
