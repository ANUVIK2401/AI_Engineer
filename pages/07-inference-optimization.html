Quantization (INT8, INT4, FP16, BF16)

[CRITICAL]
  • Quantization reduces the numeric precision used to store model weights (and sometimes activations) — from 32-bit floats down to 16-bit, 8-bit, or 4-bit representations — trading some numerical precision for smaller memory footprint and faster compute.
  • It exists because model memory footprint and memory bandwidth (not raw FLOPs) are usually the bottleneck in serving large models — halving or quartering the bytes per weight directly cuts memory movement, letting you fit bigger models on the same hardware or serve at higher throughput.
  • The insight: quantization isn't uniformly lossy across a model — outlier weight values (a small number of unusually large magnitude weights) disproportionately hurt naive quantization schemes, which is why modern quantization methods (like GPTQ, AWQ) specifically identify and handle outliers rather than uniformly rounding every weight.

▸ How this shows up in interviews
"Why does quantizing a model to INT4 sometimes barely hurt quality, and other times destroy it?"
"It depends on the model."
The impact depends heavily on the distribution of weight magnitudes and how the quantization scheme handles outliers — a small number of unusually large-magnitude weights, if naively quantized alongside everything else using a single uniform scale, force the quantization grid to be coarse everywhere to accommodate the outliers, destroying precision for the vast majority of "normal" weights. Modern quantization methods like GPTQ or AWQ specifically detect and preserve precision for these outlier-sensitive weights (or channels) while aggressively quantizing the rest, which is why naive round-to-nearest INT4 quantization can badly degrade a model while a properly outlier-aware INT4 scheme stays close to FP16 quality. This is also why quantization quality is architecture and even layer-dependent, not a single fixed number you can assume.

KV Cache & Paged Attention

[CRITICAL]
  • The KV cache stores per-token Key/Value vectors across all layers so autoregressive generation doesn't recompute attention over the full prior sequence at every step; paged attention manages this cache in fixed-size non-contiguous memory blocks, similar to OS virtual memory paging.
  • Paged attention exists because naive KV cache allocation reserves a contiguous memory block sized for the maximum possible sequence length per request, wasting huge amounts of memory when actual generations are much shorter — that waste directly limits how many requests you can batch concurrently.
  • The insight: the KV cache, not model weights, is often the actual memory bottleneck in high-throughput serving — weights are fixed size regardless of traffic, but KV cache memory scales with the number of concurrent requests × their sequence lengths, meaning serving capacity is fundamentally a memory management problem, which is exactly what vLLM's paged attention was built to solve.
▸ How this shows up in interviews
"Why does memory fragmentation matter for LLM serving throughput specifically?"
"Fragmentation wastes memory in general."
Naive KV cache allocation reserves a contiguous chunk of GPU memory sized for the worst-case (maximum) sequence length for every request, even if that particular request only generates a short response — this over-allocation means much of the reserved memory sits unused but unavailable to other requests, directly capping how many requests you can serve concurrently. Paged attention (as in vLLM) breaks the cache into small fixed-size blocks allocated on demand, similar to virtual memory paging in an OS, so memory is only consumed as actually needed and can be shared or reused across requests far more efficiently — this directly translates to serving more concurrent users on the same GPU, which is the actual throughput lever, not a minor implementation detail.

Batching Strategies

[TRADEOFF]
  • Static batching groups a fixed set of requests and waits for all to finish before starting new ones; dynamic batching adjusts batch composition between requests; continuous (in-flight) batching evicts completed sequences and admits new ones at every single decoding step.
  • These strategies exist because GPU utilization for LLM inference is maximized by keeping the batch full — but request lengths vary wildly, so a naive fixed batch wastes compute on padding once shorter sequences finish while longer ones continue.
  • The insight: continuous batching is what actually made high-throughput LLM serving practical — treating batching as a per-token decision rather than a per-request decision means a single slow, long-running request no longer blocks the GPU from picking up new, shorter requests, which is the core throughput unlock behind vLLM and similar serving frameworks.
StrategyGPU utilizationTradeoff
Static batchinglow — wastes cycles on paddingsimple, but poor throughput on variable-length outputs
Dynamic batchingmoderatebetter, but still batch-boundary bound
Continuous batchinghighbest throughput, more complex scheduler
▸ How this shows up in interviews
"Why does continuous batching improve throughput so much more than dynamic batching?"
"It processes more requests at once."
Even dynamic batching still operates at request granularity — once a batch is formed, the GPU processes it and typically doesn't add new requests mid-flight until enough of the current batch completes or a scheduling boundary is hit. Continuous batching operates at token/step granularity instead: at every single decoding step, any sequence that finished generating is evicted and a new waiting request is immediately admitted into the batch, so the GPU almost never sits idle waiting for a long-running sequence in the batch to finish. This matters enormously for LLM workloads specifically because output length is highly variable and unpredictable in advance — a single unusually long generation in a static or dynamic batch can bottleneck the throughput of an entire batch of otherwise-short requests, which continuous batching structurally prevents.

Speculative Decoding

[ADVANCED]
  • A small, fast "draft" model generates several candidate tokens ahead of time; the large target model verifies all of them in a single forward pass, accepting the ones that match what it would have generated and rejecting/regenerating from the first mismatch.
  • It exists because autoregressive decoding is inherently memory-bandwidth bound (generating one token requires one full forward pass through the entire model) — verifying multiple draft tokens in a single pass amortizes that fixed per-step cost across several tokens instead of one.
  • The insight: speculative decoding produces mathematically identical output distributions to standard decoding — it's not an approximation that trades quality for speed, it's a verification scheme that only ever accepts tokens the large model would have generated anyway, which is why it's one of the rare "free" speedups in LLM serving.
▸ How this shows up in interviews
"Doesn't using a smaller, less accurate model to draft tokens hurt output quality?"
"A little, but it's much faster so it's worth it."
It doesn't hurt quality at all, which is the key insight people often miss — the large target model still verifies every draft token against what it would have generated itself, and only accepts tokens that match; anything the draft model got wrong is rejected and regenerated by the target model directly. So the final output distribution is mathematically identical to what standard autoregressive decoding from the large model alone would have produced — the speedup comes purely from amortizing the large model's fixed per-forward-pass cost across multiple tokens when the draft model happens to guess correctly, not from any approximation or shortcut in the output itself.

Model Parallelism

[TRADEOFF]
  • Tensor parallelism splits individual weight matrices across GPUs, requiring communication within every layer's forward pass; pipeline parallelism splits different layers across different GPUs, passing activations between stages; data parallelism replicates the full model on each GPU and splits the batch of inputs across them.
  • These strategies exist because a single model can exceed a single GPU's memory capacity, and even when it fits, splitting work across GPUs can increase throughput — but each parallelism strategy has a different communication pattern and bottleneck.
  • The insight: tensor parallelism requires very high-bandwidth, low-latency interconnects (like NVLink) because it communicates within every single layer's forward pass — this is why tensor parallelism is typically limited to GPUs within the same node, while pipeline parallelism (communicating only between stage boundaries) tolerates slower interconnects and can span across nodes.

▸ How this shows up in interviews
"You have a model too large for one GPU. How do you decide between tensor and pipeline parallelism?"
"I'd use whichever framework supports it more easily."
The decision hinges heavily on interconnect bandwidth between the GPUs available — tensor parallelism requires communication within every single layer's forward and backward pass, so it needs very high-bandwidth, low-latency links like NVLink within a single node, and degrades badly over slower inter-node networking. Pipeline parallelism only communicates activations at the boundary between pipeline stages, so it tolerates slower interconnects and can span multiple nodes more gracefully, at the cost of "pipeline bubbles" — idle time while later stages wait for earlier ones to produce their first output. In practice, large-scale training and serving often combine both — tensor parallelism within a node where NVLink is available, pipeline parallelism across nodes — plus data parallelism on top for training throughput.

vLLM

[FREQUENTLY ASKED]
  • vLLM is an open-source LLM serving engine built around paged attention (block-based, on-demand KV cache management) and continuous batching, designed specifically to maximize throughput and memory efficiency for high-concurrency serving.
  • It exists because before paged attention, KV cache memory management was the dominant inefficiency in LLM serving — vLLM's specific contribution was applying an OS-style virtual memory paging concept to this problem, unlocking dramatically higher throughput on the same hardware.
  • The insight: vLLM's biggest wins come specifically from memory efficiency, not raw compute speed — it doesn't make the model's forward pass mathematically faster, it lets you fit far more concurrent requests in the same GPU memory by eliminating KV cache fragmentation and waste, which is a fundamentally different kind of optimization than quantization or kernel fusion.
▸ How this shows up in interviews
"What specific problem does vLLM solve that a naive HuggingFace Transformers serving loop doesn't?"
"vLLM is just faster and more optimized."
A naive serving loop typically allocates a contiguous KV cache block sized for the maximum sequence length per request, which wastes enormous amounts of GPU memory when actual generations are much shorter than the worst case — this memory waste directly limits how many concurrent requests you can serve. vLLM's paged attention breaks the KV cache into small fixed-size blocks allocated on demand, analogous to OS virtual memory paging, essentially eliminating that waste and letting you pack dramatically more concurrent sequences into the same GPU memory. Combined with continuous batching, which keeps the GPU fed with new requests as soon as any sequence in the current batch finishes, vLLM's throughput advantage over naive serving comes primarily from memory efficiency and scheduling, not from making the underlying model computation itself faster.

Serving Frameworks (vLLM vs TGI vs Triton)

[TRADEOFF]
  • vLLM focuses on high-throughput LLM-specific serving via paged attention; TGI (Text Generation Inference, Hugging Face) offers similarly optimized LLM serving tightly integrated with the HF ecosystem; NVIDIA Triton is a general-purpose multi-framework model server supporting arbitrary model types beyond LLMs, with more configurable deployment options.
  • Multiple frameworks exist because teams have different constraints: some need the highest raw LLM throughput, some need tight integration with an existing model hub/ecosystem, and some need a single unified serving layer across many model types (not just LLMs) in a broader ML platform.
  • The insight: the "best" serving framework depends on organizational scope as much as raw benchmarks — a team serving only LLMs typically gets the most throughput-per-dollar from vLLM specifically, but a platform team serving dozens of heterogeneous model types (vision, tabular, LLM) often standardizes on Triton for one consistent operational surface even at some LLM-specific throughput cost.

▸ How this shows up in interviews
"Your company already serves vision and tabular models on Triton. Would you still introduce vLLM just for the new LLM feature?"
"Yes, always use the best tool for each specific job."
It depends on how much the LLM-specific throughput gain actually matters for the use case versus the operational cost of maintaining a second serving stack — if the LLM feature has serious latency/cost requirements at meaningful scale, vLLM's paged attention and continuous batching can offer a large enough throughput advantage to justify the added operational surface of running two serving systems. But if the LLM workload is lower-volume or latency-tolerant, I'd lean toward keeping everything on the existing Triton deployment (which does support LLM backends) purely to avoid duplicating monitoring, deployment pipelines, and on-call expertise across two systems — the "best tool per job" instinct has to be weighed against real operational fragmentation cost, not applied automatically.

Latency vs Throughput (TTFT, TBT)

[CRITICAL]
  • Time to First Token (TTFT) measures how long a user waits before seeing any output; Time Between Tokens (TBT, or inter-token latency) measures the streaming generation speed after that; throughput measures total tokens generated per second across all concurrent requests on the system.
  • These are distinct metrics because optimizing for one can directly hurt another — maximizing system-wide throughput via large batches can increase individual-request TTFT, since a request might wait for a batch slot or share compute with many concurrent requests.
  • The insight: users experience latency (TTFT, TBT) but the business pays for throughput (tokens/sec/GPU) — these two are frequently in tension, and the "right" operating point is a product decision about acceptable user-perceived latency, not a pure engineering optimization target.
Situation
What breaks or costs
Maximizing batch size for throughput
Individual request TTFT increases — user-perceived latency suffers
Minimizing latency via small batches
GPU underutilized — throughput and cost-per-token both worsen
▸ How this shows up in interviews
"Your product team wants both lower latency and lower cost per request. How do you explain the tension?"
"We just need better hardware."
Lower cost per request generally comes from higher GPU utilization, which means larger batches and more concurrent requests sharing the same compute — but that same batching is exactly what increases individual-request latency, since a request may need to wait for a batch slot or share forward-pass compute with many others. Continuous batching and paged attention push this Pareto frontier outward (better throughput at the same latency, or vice versa) but they don't eliminate the fundamental tradeoff — at some point, further throughput gains require accepting worse tail latency for individual users. The real conversation with product is about defining an acceptable latency SLA (e.g., "TTFT under 500ms for 95% of requests") and then optimizing cost-per-token within that constraint, rather than treating latency and cost as independently optimizable targets.
