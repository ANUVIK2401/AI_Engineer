Attention Mechanism (Q/K/V)

[CRITICAL]
  • Each token produces a Query, Key, and Value vector; attention scores each token pair via the dot product of Query and Key, softmaxes those scores, and uses them to weight a sum of Value vectors.
  • This exists to let every token dynamically gather information from every other token in the sequence, weighted by relevance — solving the fixed-context bottleneck that plagued RNNs.
  • The insight: Q, K, and V are all linear projections of the same input embeddings using different learned weight matrices — the "roles" of query/key/value aren't inherent to the token, they're entirely learned during training to make retrieval-like lookups useful for the task.
$$ \text{Attention}(Q,K,V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V $$
▸ How this shows up in interviews
"Walk me through scaled dot-product attention end to end."
"It computes attention scores and applies them to the values."
Each token's embedding is linearly projected into three vectors — Query, Key, Value — using separate learned weight matrices. For every pair of tokens, we compute the dot product of one token's Query with another's Key, producing a raw relevance score; we scale these by 1/√d_k to keep gradients stable, then softmax across each row so scores for a given token sum to 1. Finally, each token's output is a weighted sum of all tokens' Value vectors, weighted by those softmax scores — meaning the token "looks at" the rest of the sequence and pulls in information proportional to computed relevance.
Follow-up: "Why do we need separate Query and Key projections instead of just using the raw embeddings directly?"

Multi-Head Attention

[FREQUENTLY ASKED]
  • Multi-head attention runs several attention operations in parallel, each with its own learned Q/K/V projections into a smaller subspace, then concatenates and projects the results back to full dimension.
  • It exists because a single attention head can only learn one type of relationship pattern per layer — multiple heads let the model attend to different relationships (syntax, coreference, position) simultaneously.
  • The insight: heads aren't explicitly assigned roles during training — interpretability research shows some heads emerge specializing in syntax, others in positional patterns, purely as a byproduct of gradient descent optimizing the loss, not by design.
▸ How this shows up in interviews
"Why use multiple attention heads instead of one larger head?"
"More heads means the model can learn more things."
A single attention head produces one weighted average over the sequence per token, which forces every type of relationship (syntactic, semantic, positional) to be blended into a single attention pattern — that's a representational bottleneck. Splitting into multiple smaller-dimensional heads lets each specialize in a different relationship type and computes them in parallel at roughly the same total computational cost as one large head, since the per-head dimension shrinks proportionally. Empirically, some heads do specialize (e.g., attending to the previous token, or matching coreferent noun phrases) which validates the architectural intuition even though it emerges rather than being designed in.
Follow-up: "Does adding more heads always help, or is there a point of diminishing returns?"

Positional Encoding

[ADVANCED]
  • Absolute positional encoding adds a fixed or learned vector per position to token embeddings; relative encoding injects position information into the attention computation itself; RoPE (Rotary Position Embedding) rotates Q/K vectors by an angle proportional to position.
  • Positional encoding exists because self-attention alone is permutation-invariant — without it, the model can't distinguish token order at all.
  • The insight: RoPE's rotation trick means relative position is naturally encoded in the dot product between rotated Q and K vectors — it extrapolates to longer sequences at inference than were seen during training far better than fixed absolute encodings, which is exactly why it's the default choice for modern long-context LLMs (LLaMA, GPT-NeoX).

▸ How this shows up in interviews
"Why does RoPE generalize to longer contexts better than learned absolute positional embeddings?"
"It's a newer, better technique."
Learned absolute positional embeddings are a fixed lookup table indexed by position, so any position beyond the max length seen in training simply has no learned embedding — it can't extrapolate at all without retraining. RoPE instead encodes position by rotating the Query and Key vectors by an angle proportional to their position, and crucially the attention score between two tokens ends up depending only on their relative distance, not absolute position, because the rotation is baked directly into the dot product math. This relative-distance property is what lets RoPE-based models handle longer sequences at inference than trained on, sometimes assisted by additional interpolation/scaling techniques for very large extensions.

Transformer Block Anatomy

[FREQUENTLY ASKED]
  • Each transformer block runs: multi-head self-attention → residual add + layer norm → feed-forward network (two linear layers with a nonlinearity) → residual add + layer norm.
  • This specific structure exists because attention mixes information across tokens (a "communication" step) while the feed-forward network processes each token independently (a "computation" step) — alternating them lets the model both gather context and transform it.
  • The insight: the feed-forward network, not attention, holds the majority of a transformer's parameters (typically ~⅔) — most of what people call "the attention model" is actually per-token MLP computation, with attention serving mainly as the mechanism for moving information between positions.
▸ How this shows up in interviews
"Why is there a residual connection around both the attention and FFN sublayers?"
"It helps with gradient flow."
Residual connections let each sublayer learn a correction to the identity function rather than the entire transformation from scratch, which makes very deep stacks (dozens to over a hundred layers) actually trainable — without them, gradients would have to flow through every nonlinear transform at every layer, compounding vanishing gradient effects. Concretely, the output is x + Sublayer(LayerNorm(x)), so even if a particular sublayer contributes very little useful signal at some point in training, the residual path preserves the original information rather than losing it, which stabilizes training dramatically at transformer scale.
Follow-up: "What's the difference between Pre-LN and Post-LN transformer variants, and why do most modern LLMs use Pre-LN?"

Decoder-Only vs Encoder-Decoder

[FREQUENTLY ASKED]
  • Decoder-only models (GPT family) use causal (masked) self-attention so each token only attends to previous tokens, trained purely on next-token prediction; encoder-decoder models (T5) use a bidirectional encoder to process the full input, then a decoder that cross-attends to the encoder's output.
  • Both exist because different tasks benefit from different information flow — open-ended generation benefits from a single unified autoregressive objective, while tasks with a clear distinct input/output structure (translation, summarization) can benefit from an encoder that sees the full input bidirectionally first.
  • The insight: decoder-only architectures won out for general-purpose LLMs not because bidirectional context is useless, but because a single unified autoregressive objective scales more simply across arbitrarily many tasks framed as text generation — you don't need task-specific encoder/decoder architecture decisions, just more data and parameters.
▸ How this shows up in interviews
"Why did GPT-style decoder-only models become the dominant architecture over encoder-decoder models like T5?"
"Decoder-only is simpler."
Decoder-only models unify every task — translation, summarization, question answering, chat — under a single training objective (next-token prediction), so scaling the model and data doesn't require redesigning an architecture per task, unlike encoder-decoder setups that need task-specific framing. This uniformity turned out to matter more than the theoretical benefit of bidirectional encoding for input understanding, because in-context learning and instruction-following emerged from scale on the decoder-only objective alone. Encoder-decoder models still have real advantages for tasks with clearly asymmetric input/output structure and fixed-length outputs (like classic machine translation), which is why they haven't disappeared entirely.

Tokenization (BPE, WordPiece)

[FREQUENTLY ASKED]
  • Byte-Pair Encoding (BPE) and WordPiece build a vocabulary by iteratively merging the most frequent adjacent character/subword pairs, producing subword units instead of full words or raw characters.
  • Subword tokenization exists to balance vocabulary size against sequence length — word-level tokenization can't handle unseen words (out-of-vocabulary), while character-level tokenization makes sequences impractically long.
  • The insight: vocabulary size is a direct tradeoff against sequence length and embedding table size — a larger vocab means shorter sequences per input (cheaper attention, which scales quadratically) but a bigger embedding/output matrix, so vocab size is a deliberate engineering choice, not just "bigger is better."
▸ How this shows up in interviews
"Why do LLMs struggle with tasks like counting letters in a word or reversing a string?"
"Because they're language models, not calculators."
The model never actually sees individual characters — tokenization collapses common words and subwords into single tokens, so a word like "strawberry" might be a single opaque token or split into two or three chunks that don't align with individual letters at all. Since the model's entire input representation operates at the token level, it has no direct access to character-level structure unless that structure happened to be reflected in how the tokenizer split similar words during its own training — this is a tokenization artifact, not a reasoning limitation, and it's why prompting tricks like spelling a word letter-by-letter with spaces can improve performance on these tasks.
Follow-up: "How does vocabulary size affect training and inference cost?"

Context Window & KV Cache

[CRITICAL]
  • The context window is the maximum number of tokens a model can attend to at once; the KV cache stores each generated token's Key and Value vectors so they don't need to be recomputed for every new token during autoregressive generation.
  • The KV cache exists because without it, generating each new token would require recomputing attention over the entire prior sequence from scratch — caching turns that into an incremental append.
  • The insight: attention cost scales quadratically with sequence length (every token attends to every other token), which is why context window increases are far more expensive than linear — doubling context roughly quadruples the attention compute and KV cache memory footprint, which is the real bottleneck behind long-context serving costs.

▸ How this shows up in interviews
"Why is serving a 100K-token context so much more expensive than a 10K-token one — isn't it just 10x?"
"Longer context needs more memory."
It's worse than linear on two fronts: attention computation scales quadratically with sequence length since every token must compute a score against every other token, so 10x the tokens means roughly 100x the attention FLOPs for that operation. Additionally, the KV cache itself grows linearly with sequence length per request, and since it must be held in GPU memory for the duration of generation, longer contexts directly reduce how many concurrent requests you can batch — hurting both compute cost and serving throughput simultaneously, which compounds why long-context serving is disproportionately expensive.
Follow-up: "What is paged attention and what specific inefficiency does it solve?"

Temperature, Top-p, Top-k Sampling

[FREQUENTLY ASKED]
  • Temperature rescales logits before softmax to control randomness; top-k restricts sampling to the k highest-probability tokens; top-p (nucleus sampling) restricts to the smallest set of tokens whose cumulative probability exceeds p.
  • These exist because always picking the single most likely token (greedy decoding) produces repetitive, low-diversity text, while pure random sampling from the full distribution produces incoherent text by occasionally picking very low-probability tokens.
  • The insight — the classic gotcha: greedy decoding is deterministic and often produces worse quality output than sampling with a moderate temperature, because always taking the locally most-probable token can lead down a path that's globally suboptimal — it's a form of exposure bias, not a "safe default."
▸ How this shows up in interviews
"Why wouldn't you always use greedy decoding if you want the 'best' possible output?"
"Greedy decoding is more repetitive."
Greedy decoding picks the single highest-probability token at each step, which is a locally optimal but not globally optimal strategy — an early token choice can lock the model into a lower-quality continuation it can't recover from, since there's no lookahead or backtracking. It's also degenerate in a specific way: it tends to produce repetitive loops, because once a phrase becomes locally likely, greedy decoding gets stuck re-selecting it. This is why techniques like beam search (limited lookahead) or sampling with temperature/top-p (controlled randomness) generally produce more natural, higher-quality text than pure greedy decoding despite being the "safe-looking" deterministic choice.
Follow-up: "How would you tune temperature and top-p differently for a code-generation task versus a creative-writing task?"

RLHF

[CRITICAL]
  • RLHF trains a reward model on human preference comparisons between model outputs, then uses that reward model to fine-tune the LLM via reinforcement learning (typically PPO), optimizing generations to score highly under the learned reward.
  • It exists because "good response" isn't a label you can directly supervise — there's no ground truth token sequence for "be helpful and harmless," so you instead learn a proxy reward function from relative human judgments and optimize against that.
  • The insight: RLHF doesn't teach the model new knowledge or capabilities — pretraining already encodes that. It reshapes which parts of the model's existing distribution get sampled, steering output style, refusals, and helpfulness — which is why RLHF'd models can still hallucinate confidently on facts they were never taught to verify.
▸ How this shows up in interviews
"What problem does RLHF actually solve that supervised fine-tuning alone doesn't?"
"It makes the model more aligned with human values."
Supervised fine-tuning requires a fixed target sequence to imitate, but for open-ended qualities like "helpful, harmless, and honest" there's often no single correct response — instead there's a preference ranking between candidate responses that's much easier for humans to provide than writing a gold-standard answer. RLHF converts that pairwise preference signal into a learned reward model, then optimizes the policy (the LLM) against it via PPO, which lets you optimize for a nuanced, hard-to-specify objective that pure supervised learning on fixed targets can't directly express. In practice it also introduces real risks — reward hacking, where the policy finds ways to score high on the reward model without actually being better, which is why techniques like KL-penalty regularization against the original SFT model are used to keep the policy from drifting too far.
Follow-up: "What is reward hacking and how does the KL penalty in PPO help prevent it?"

Fine-Tuning Types (LoRA, QLoRA)

[CRITICAL]
  • Full fine-tuning updates all model parameters; LoRA freezes the base model and injects small trainable low-rank matrices into attention layers; QLoRA does the same but on a quantized (4-bit) frozen base model; prefix tuning prepends trainable "virtual tokens" without touching model weights at all.
  • Parameter-efficient methods exist because full fine-tuning of a multi-billion parameter model requires prohibitive GPU memory (for both weights and optimizer states) that most teams can't afford for every downstream task.
  • The insight: LoRA works because weight updates during fine-tuning empirically have low "intrinsic rank" — the actual change needed to adapt a pretrained model to a new task lives in a much lower-dimensional subspace than the full parameter space, so you can approximate it with two small matrices instead of updating the whole weight matrix.

▸ How this shows up in interviews
"When would you choose full fine-tuning over LoRA despite the cost?"
"When you have enough GPU budget."
LoRA assumes the needed weight update has low intrinsic rank — true for most task adaptation, like teaching a model a new tone or domain vocabulary — but if the target task requires a substantial distributional shift from the pretraining data (a wildly different language, or a completely different modality-adjacent task), the low-rank approximation may not have enough capacity to represent the needed change, and full fine-tuning captures higher-rank updates LoRA structurally cannot. In practice, I'd start with LoRA/QLoRA as the default given the massive cost savings, and only justify full fine-tuning with an ablation showing LoRA underperforms by a meaningful margin on held-out eval — not assume it upfront.
