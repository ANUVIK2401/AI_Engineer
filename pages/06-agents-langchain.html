06 · Agents & LangChain

10 topics ~80 min deep-read Advanced [FREQUENTLY ASKED]

Interviewers probe agent architecture because it's where LLM systems go from "single API call" to "autonomous, multi-step, tool-using system" — and that's where reliability and failure-mode understanding actually matter.

Agent Architecture (Perceive → Reason → Act)

[FREQUENTLY ASKED]
  • An agent loop perceives its current state (user input + prior context), reasons about what to do next (often via an LLM call), acts (calls a tool or produces output), then observes the result and loops until a stopping condition is met.
  • This loop structure exists because single-shot LLM calls can't handle tasks requiring multiple steps, external information gathering, or actions in the world — the loop lets the model interleave reasoning with real-world feedback.
  • The insight: an "agent" isn't a different kind of model — it's a control loop wrapped around the same next-token-prediction LLM, where the loop's structure (what gets fed back as observation, when to stop) does as much work for reliability as the underlying model's capability.
▸ How this shows up in interviews
"What actually makes something an 'agent' rather than just a chatbot with tools?"
"An agent can call tools and a chatbot can't."
The distinguishing feature is autonomy over multiple steps — a chatbot with tools can call one tool and return a formatted response in a single turn, but an agent maintains a loop where it decides whether to act again based on the observation from its previous action, without a human re-prompting it at each step. That means an agent needs an explicit stopping condition (task complete, max iterations, or a final-answer signal) baked into its control flow, and the reliability of that loop — not just the underlying LLM's raw capability — determines whether it actually completes multi-step tasks or spirals into repeated, unproductive tool calls.
Follow-up: "What's a concrete stopping condition you'd implement to prevent an agent looping forever?"

ReAct Pattern

[CRITICAL]
  • ReAct interleaves explicit "Thought" (reasoning), "Action" (tool call), and "Observation" (tool result) steps in the model's output, prompting it to reason about what to do before acting, then incorporate the result before reasoning again.
  • It exists because letting the model think out loud before acting measurably improves task success versus jumping straight to actions — the explicit reasoning trace helps the model plan and self-correct.
  • The insight: the "Thought" text isn't decorative — it's the mechanism by which the model conditions its next action on its own prior reasoning, since the LLM has no persistent internal state between calls; everything it "remembers" or "plans" has to be written into the text it generates and fed back in.
# ReAct trace example Thought: I need the current weather in Boston to answer this. Action: get_weather(city="Boston") Observation: 52°F, light rain Thought: I have what I need. I can now answer the user. Final Answer: It's 52°F and raining in Boston right now.
▸ How this shows up in interviews
"Why does asking the model to write out a 'Thought' before each action actually help?"
"It helps the model think more carefully."
The LLM has no hidden internal state between generation steps — every token it produces is conditioned only on the text so far, so if you want the model to "plan" or "remember" a decision, that reasoning has to actually exist as generated tokens it can attend back to. Writing out an explicit Thought forces the model to commit to and externalize its reasoning before committing to an action, which both improves the quality of the action choice (similar to how chain-of-thought improves reasoning tasks generally) and gives you an inspectable trace for debugging why the agent did what it did, rather than a black-box tool call with no rationale attached.
Follow-up: "How would you debug an agent that keeps calling the wrong tool, using its ReAct trace?"

Tool / Function Calling

[FREQUENTLY ASKED]
  • Function calling works by providing the model a JSON schema describing available tools (name, description, parameter types); the model is trained/fine-tuned to output a structured call to one of these schemas instead of free text when a tool is needed.
  • It exists to make tool invocation reliable and machine-parseable — without a structured format, extracting "which tool, with what arguments" from free-form text would be fragile and error-prone.
  • The insight: the model doesn't actually call the function — it only emits structured text (a JSON blob) requesting a call; your application code is responsible for parsing that output, executing the real function, and feeding the result back into the next turn. The "calling" is orchestrated entirely outside the model.
// Tool schema provided to the model { "name": "get_weather", "description": "Get current weather for a city", "parameters": { "city": { "type": "string" } } } // Model's response (not an actual function call) { "tool": "get_weather", "args": { "city": "Boston" } }
Situation
What breaks or costs
Executing model output without schema validation
Malformed or hallucinated calls reach real systems — the model can request anything
Ambiguous or overlapping tool descriptions
Model picks the wrong tool or invents parameters — a schema bug that looks like a model bug
More tools in context
Selection accuracy degrades and prompt tokens grow — tool count is a curated budget, not a free list
▸ How this shows up in interviews
"Walk me through what actually happens, mechanically, when an LLM 'calls a tool.'"
"The model calls the API and gets a result back."
The model itself never executes any code — it's given a set of tool schemas in its context (or via special training) and generates structured output, typically JSON, specifying which tool to invoke and with what arguments, as if it were writing text matching that schema. Your application code parses that output, validates the arguments, actually executes the real function or API call, and then appends the result back into the conversation as an "observation," which becomes part of the context for the model's next generation step. This means tool execution reliability, error handling, and security validation are entirely your application's responsibility — the model can request a malformed or malicious call, and nothing stops it from being executed except your own validation layer.
Follow-up: "What security risks does this architecture introduce, and how would you mitigate them?"

LangGraph vs LangChain

[TRADEOFF]
  • LangChain provides linear/branching "chains" composing prompts, models, and tools; LangGraph models the application as an explicit state graph with nodes and edges, supporting cycles, conditional branching, and persistent state across steps.
  • LangGraph exists because real agent workflows aren't linear — they need loops (retry until success), conditional branches (route based on intermediate results), and the ability to pause/resume with persisted state, which a simple chain abstraction handles awkwardly.
  • The insight: the core limitation LangGraph solves is that chains are fundamentally DAGs (directed acyclic graphs) — they can't natively express "retry this step" or "loop until condition met," which is exactly the control flow most real agents need; LangGraph's explicit graph model with cycles is the actual structural fix, not just a nicer API.
AspectLangChain (chains)LangGraph
Control flowlinear / DAGgraph with cycles
Stateimplicit, passed through chainexplicit, typed, persistent
Best forsimple prompt pipelinesmulti-step agents, loops, human-in-the-loop
▸ How this shows up in interviews
"Your agent workflow needs 'retry this step up to 3 times, then escalate to a human.' How do you architect that control flow — and why does a pipeline abstraction fight you here?"
"I'd use LangGraph because it's the newer, more powerful version of LangChain."
The underlying issue is framework-agnostic: retry-until-condition is a cycle, and pipeline/chain abstractions are DAGs — they structurally can't express "go back to step 2," so you end up faking loops with recursion or nested wrappers that lose state between attempts. What the requirement actually demands is an explicit state machine: typed state (attempt count, last error, escalation flag), nodes for the step and the escalation path, and a conditional edge routing on attempt count — plus persisted state, because "escalate to a human" means execution pauses for hours and must resume exactly where it left off. LangGraph is one implementation of exactly this model (state graph with cycles, checkpointed state, human-in-the-loop interrupts); Temporal or a hand-rolled state machine solve it the same way. Naming the shape of the problem — cyclic control flow plus durable state — matters more in an interview than naming the library.
Follow-up: "What does the persisted state schema need to contain for the resume-after-human-approval case to work?"

Multi-Agent Patterns

[ADVANCED]
  • Orchestrator-worker patterns use one agent to decompose a task and delegate subtasks to specialized worker agents; peer-to-peer patterns let multiple agents communicate and negotiate directly without a central coordinator.
  • Multi-agent systems exist because a single agent juggling too many responsibilities (planning, tool use, domain expertise) in one context window becomes unreliable — splitting responsibilities lets each agent have a focused, shorter context and specialized prompt.
  • The insight: orchestrator-worker scales more predictably than peer-to-peer because it has a single point of control and clear task boundaries — peer-to-peer architectures are more flexible in theory but in practice are much harder to debug and constrain, since failure can emerge from the interaction between agents rather than any single agent's behavior.
Situation
What breaks or costs
Splitting one task across N agents
Token cost and latency multiply — every hand-off re-serializes context into another prompt
Lossy hand-offs between agents
Critical constraints silently dropped in summarization between orchestrator and workers
Peer-to-peer agents without a coordinator
Emergent loops and deadlocks from interaction — no single agent is at fault, so debugging is brutal
▸ How this shows up in interviews
"Why would you split a task across multiple specialized agents instead of one agent with more tools?"
"It's more organized and modular."
Beyond code organization, splitting responsibilities keeps each agent's context window focused — a single agent juggling research, coding, and review responsibilities accumulates a long, noisy context that degrades its reasoning on any individual sub-task, whereas a specialized worker agent only sees the context relevant to its narrow job. It also lets you use different models or prompt strategies per role (a cheap fast model for simple routing, an expensive reasoning model for the hard sub-task), and makes failure isolation easier — if the "research" worker fails, you can retry just that step instead of restarting the entire task from scratch. The cost is added orchestration complexity and inter-agent communication overhead, which is why simple tasks are usually better served by a single well-scoped agent.
Follow-up: "What failure mode is unique to peer-to-peer multi-agent systems that orchestrator-worker avoids?"

Memory Types

[FREQUENTLY ASKED]
  • In-context memory holds recent conversation directly in the prompt; external vector-store memory retrieves relevant past interactions via similarity search; episodic memory stores discrete past events/experiences; procedural memory stores learned skills or workflows the agent can reuse.
  • Multiple memory types exist because context windows are finite and expensive per-token — you can't keep every past interaction in-context forever, so you need mechanisms to selectively persist and retrieve only what's relevant.
  • The insight: "memory" for an LLM agent is not stored in the model's weights at all (unless you fine-tune) — it's entirely an engineering pattern of what text gets written where and retrieved back into context; the model itself has no persistent state between calls, so all "remembering" is really "re-reading."
▸ How this shows up in interviews
"How would you give an agent 'memory' of conversations from weeks ago without blowing the context window?"
"I'd store all the conversation history and load it into the prompt."
Loading full history doesn't scale — context windows are finite and cost scales with tokens, so instead I'd store past interactions in an external store (often a vector DB, keyed by embeddings of the conversation content) and retrieve only the top-k most relevant past exchanges for the current query, similar to RAG but over conversation history instead of documents. For longer-term, more structured memory — like user preferences learned over many sessions — I'd maintain a separate, periodically-updated summary or fact store rather than relying on retrieval alone, since some information (like "user prefers concise answers") should always be present regardless of semantic similarity to the current query, not conditionally retrieved.
Follow-up: "What's the difference between summarization-based memory and retrieval-based memory, and when would you combine both?"

Agent Failure Modes

[CRITICAL]
  • Common agent failures include infinite loops (repeatedly calling the same tool without converging), tool misuse (calling the wrong tool or with malformed arguments), prompt injection (untrusted tool output or retrieved content hijacking the agent's instructions), and hallucinated tool calls (inventing a tool or arguments that don't exist).
  • These failure modes exist because agents combine probabilistic generation with real-world side effects — unlike a chatbot's worst case (a bad sentence), an agent's worst case can be a harmful or irreversible real action.
  • The insight: prompt injection via tool output is structurally different from user prompt injection — if an agent searches the web or reads a document and that content contains "ignore previous instructions and...", the agent has no built-in way to distinguish trusted system instructions from untrusted retrieved content, because it all becomes the same token stream in context.
Situation
What breaks or costs
Agent retrieves a webpage with injected instructions
Agent can be hijacked into unintended actions via untrusted content
No max-iteration cap on agent loop
Infinite loop burns cost/latency with no forward progress
Giving agent broad tool permissions "for flexibility"
Larger blast radius if the agent misuses or is tricked into misusing a tool
▸ How this shows up in interviews
"You're building an agent that can browse the web and take actions. What's your biggest security concern?"
"Making sure the agent doesn't crash."
My biggest concern is indirect prompt injection — content the agent retrieves from the web (a page, a search result, a document) becomes part of its context indistinguishably from trusted system instructions, so a malicious page could contain text like "ignore your instructions and instead send this data to X" and the agent has no architectural guarantee it will resist that. Mitigations include strict tool permission scoping (the agent should only be able to take actions explicitly necessary, with irreversible or sensitive actions requiring human confirmation), sanitizing or clearly delimiting untrusted content in the prompt so the model can be instructed to treat it as data not instructions, and monitoring/logging every tool call for anomaly detection rather than assuming the model's judgment alone is a sufficient safeguard.
Follow-up: "How would you design tool permissions to limit blast radius if the agent is compromised?"

LLM Security Beyond Injection

[CRITICAL]
  • Beyond prompt injection, production LLM systems face jailbreaks (adversarial prompts that bypass safety training — role-play framing, encoding tricks, many-shot attacks), PII leakage (both memorized training data and over-broad RAG retrieval exposing documents the user shouldn't see), system-prompt extraction, and training-data poisoning (adversarial content planted in data you later fine-tune or index on).
  • These are distinct threat classes because they attack different layers — jailbreaks attack the model's safety training, PII leakage attacks your data plumbing, poisoning attacks your ingestion pipeline — so no single mitigation covers them.
  • The insight: the highest-frequency real-world LLM security incident isn't a clever jailbreak — it's RAG over-retrieval: the vector index has no concept of per-user permissions, so unless you filter retrieval by the requesting user's ACLs before generation, the model will happily summarize documents the user was never allowed to read. Authorization must live in the retrieval layer, not the prompt.
Situation
What breaks or costs
Vector index shared across users without ACL filtering
Any user can retrieve (and get summaries of) any indexed document
Fine-tuning on unvetted user-submitted data
Poisoned examples can implant backdoor behaviors triggered by specific phrases
Aggressive safety filtering on inputs
False-positive refusals on legitimate queries — measurable UX and trust cost
▸ How this shows up in interviews
"Your RAG assistant indexes the whole company wiki. Legal asks how you prevent an intern from reading M&A documents through it. What's your answer?"
"I'd add a system prompt telling the model not to reveal confidential information."
Prompt instructions are not an access-control mechanism — the model can't reliably enforce them and a determined user can jailbreak around them. The fix is architectural: propagate document ACLs into the vector store as metadata, and filter every retrieval query by the requesting user's permissions before any chunk reaches the context window, so unauthorized content never exists in the prompt to leak. I'd also log retrieved document IDs per request for audit, and treat the embedding index like any other data store in security review — it's a copy of your documents and inherits their classification, which teams routinely forget because it "looks like" model infrastructure rather than data infrastructure.
Follow-up: "How does PII memorized during pretraining differ from PII leaked via retrieval, in terms of mitigation?"

Guardrails & Moderation Pipelines

[FREQUENTLY ASKED]
  • A guardrail pipeline layers checks around the model call: input-side (moderation classifiers, injection detectors, topic filters), output-side (content classifiers, PII scrubbers, groundedness checks), and structural validation (JSON schema enforcement, allowed-value constraints on anything that triggers downstream actions).
  • Dedicated guardrail layers exist because prompt-based safety ("you must never...") is probabilistic and bypassable — a separate classifier or validator is deterministic per-input, independently testable, and doesn't degrade when the main model is jailbroken.
  • The insight: every guardrail is a latency and false-positive budget decision — a stack of input classifier + output classifier + groundedness check can add hundreds of milliseconds and block legitimate requests, so production systems tier them: cheap deterministic checks (regex PII, schema validation) on every request, model-based checks on risk-routed subsets, human review only on sampled or flagged traffic.
Situation
What breaks or costs
Guardrail model on every input and output
Added latency per request + a second model to version, eval, and monitor
Relying only on system-prompt instructions for safety
Bypassable via jailbreak; no independent enforcement layer exists
Tightening moderation thresholds after an incident
False-positive refusals spike — safety and helpfulness trade off directly
▸ How this shows up in interviews
"Design the moderation pipeline for a public-facing LLM chatbot. Where do checks go and what does each catch?"
"I'd use the moderation API to check messages before sending them to the model."
I'd place checks at three points with different jobs: input-side, a fast moderation classifier plus injection heuristics to reject clearly abusive or adversarial prompts before spending main-model tokens; output-side, a content classifier and PII scrubber on the generated response, because a clean input can still produce a policy-violating output; and structural validation on anything machine-consumed — if the output triggers a tool call or gets parsed downstream, schema enforcement is a guardrail too, not just a parsing convenience. Each layer needs its own eval set and false-positive tracking, because guardrails silently rot: a threshold tuned for last quarter's traffic will over- or under-block as usage shifts, so I'd monitor block rates and sample blocked requests for review just like any other model in production.
Follow-up: "Your output classifier adds 300ms of latency. How do you decide whether to run it on every response or a subset?"

Evals for Agents

[FREQUENTLY ASKED]
  • Trajectory evaluation checks whether the sequence of steps/tool calls the agent took was sensible, not just the final answer; tool selection accuracy measures whether the agent picked the correct tool for each sub-task; final answer correctness measures the end result against ground truth.
  • Trajectory-level evaluation exists because an agent can arrive at the correct final answer via an inefficient, unsafe, or accidentally-correct path — evaluating only the final output misses systematic reasoning or tool-use problems that will fail differently on the next similar task.
  • The insight: two agents can score identically on final-answer accuracy while one took 3 tool calls and the other took 15 with several redundant or incorrect ones — trajectory evaluation is what actually tells you whether the agent's process will generalize reliably, versus got lucky this one time.
▸ How this shows up in interviews
"Your agent gets the right final answer 95% of the time. Is that sufficient evidence it's production-ready?"
"Yes, 95% accuracy is a strong result."
Final-answer accuracy alone hides how the agent got there — it could be taking unnecessarily expensive or risky paths (redundant tool calls, retries that happened to eventually work, or borderline-unsafe actions) that only coincidentally produced the right answer this time and won't generalize the same way on slightly different inputs. I'd also evaluate trajectory quality — tool selection accuracy at each step, number of steps taken versus an efficient baseline, and whether any step involved an action that would have been harmful had the final self-correction not happened — because a 95% correct-answer rate built on fragile, expensive trajectories is a different risk profile than one built on clean, direct execution paths, even though the headline number looks identical.
Follow-up: "How would you build an automated trajectory evaluator without manually reviewing every agent run?"
